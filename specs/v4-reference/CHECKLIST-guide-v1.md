# Чек-лист проверки руководства целиком (v1)

> **Назначение:** агент Claude Code (или живой ревьюер) применяет этот чек-лист к готовому руководству — перед merge в `main` или при ежеквартальном content-аудите.
>
> **Применяется к:** одному из 4 руководств v4 — целиком, не к разделу и не к подразделу.
>
> **Когда запускать:**
> 1. После завершения нового руководства (полный проход PD.FORM.103 1→10 по всем S/SS)
> 2. Ежеквартально на стабильных руководствах (content-аудит)
> 3. После крупного изменения Pack (PD.FORM.089 v4 → v5, например)
>
> **Уровни:** 🔴 машина / 🟡 агент / 🟢 пилот. Правило вложенности: 🔴 100% PASS → 🟡 100% PASS → 🟢 ≥3 пилота × 6/6.
>
> **Иерархия чек-листов:** руководство (этот) → раздел (CHECKLIST-section-v1.md) → подраздел (CHECKLIST-subsection-v1.md). Этот чек-лист НЕ дублирует проверки на уровне раздела/подраздела — он добавляет проверки, специфичные для руководства целиком.

## Уровень 🔴: Машинная проверка (v4-lint guide-mode)

> Запускается через `python3 tools/v4-lint.py guide <guide-id>`. Exit 0 = PASS.

### 🔴 A. Структурная полнота руководства

```bash
python3 tools/v4-lint.py guide --id <guide-id> specs/v4-reference/
```

- [ ] **A.1 Все S заполнены:** S1...SN присутствуют (по `structure-guide-N.md`), нет пропусков
- [ ] **A.2 Все SS заполнены:** для каждого S — SS1...SSM или явный `optional: true`
- [ ] **A.3 Frontmatter руководства:** есть `guide_id`, `title`, `object`, `axis`, `stages` (ст. от-до), `last_updated`
- [ ] **A.4 README руководства:** есть `README.md` с навигацией по разделам
- [ ] **A.5 Версия руководства:** в `version.json` зафиксирована (semver `v4.X.Y`)

### 🔴 B. Кросс-руководная согласованность

```bash
python3 tools/v4-lint.py cross-guide --scope guide <guide-id> specs/v4-reference/
```

- [ ] **B.1 Понятия не дублируются между руководствами:** одно понятие = одно руководство-источник (introduces), остальные используют (uses)
- [ ] **B.2 Bounded Context руководства соблюдён:** `mastery_node` всех подразделов попадает в зону руководства (A.1.1)
- [ ] **B.3 Нет orphan-ссылок:** каждый `uses` в подразделе ссылается на существующий `introduces` (в этом или другом руководстве)
- [ ] **B.4 Pack-маппинг полон:** все понятия из `introduces` всех подразделов имеют ссылку на Pack (`PD.FORM/METHOD/CAT.NNN`)

### 🔴 C. Граф понятий руководства

```bash
python3 tools/v4-lint.py graph build --scope guide <guide-id> --out-json /tmp/guide-graph.json
```

- [ ] **C.1 Ациклический граф:** в графе понятий нет циклов
- [ ] **C.2 Связность:** ≥80% понятий имеют ≥1 входящее или исходящее ребро (нет «островов»)
- [ ] **C.3 Соответствие prerequisites:** каждый `prerequisites` в SS существует ДО SS в порядке чтения

### 🔴 D. Pack-drift (для руководства целиком)

```bash
python3 tools/v4-lint.py pack-drift --scope guide <guide-id> --pack PACK-personal/...
```

- [ ] **D.1 Все cp.* / bh.* актуальны:** соответствуют текущему PD.FORM.089
- [ ] **D.2 Все PD.FORM/METHOD/CAT/FAIL.NNN валидны:** ссылки разрешаются в Pack
- [ ] **D.3 Нет deprecated-понятий:** ни один `introduces` не помечен `deprecated` в Pack

## Уровень 🟡: Агент-проверка (Claude Opus, context isolation)

> Субагент читает руководство целиком (все S + все SS) и применяет содержательные критерии. Запускается через Task tool с context isolation.

### 🟡 E. Целостность объекта руководства

- [ ] **E.1 Объект ясен с первого раздела:** в S1 явно сказано «это руководство о Пилоте/Машине/инструменте мышления»
- [ ] **E.2 Объект сохраняется во всех разделах:** ни в одном S/SS объект не подменяется (например, в руководстве о Пилоте не появляются разделы об IWE как самоцели)
- [ ] **E.3 Границы соблюдены:** ни один S не залезает в зону другого руководства (см. различение 4 руководств в WP-300)

### 🟡 F. Дуга нарратива по всему руководству

- [ ] **F.1 Прогрессия по ступеням:** S1-Sk охватывают ст. 1-2 (вход, конкретика), Sk-SN — ст. 3-5 (горизонт, системность)
- [ ] **F.2 Нет «прыжков» через ступени:** между соседними S нет разрыва — каждый следующий S опирается на предыдущий
- [ ] **F.3 Мировоззренческий сдвиг отчётлив:** к концу руководства читатель должен видеть мир иначе, чем в начале — этот сдвиг сформулирован явно (в финальном S или эпилоге)
- [ ] **F.4 Тональность согласована с дугой:** ст. 1-2 разделы — бытовой язык, конкретные первые шаги; ст. 3-5 разделы — системные понятия, связь с надсистемой

### 🟡 G. Охват узлов мастерства

- [ ] **G.1 Узел руководства покрыт:** все cp/bh-измерения узла (по PD.FORM.089) затронуты хотя бы в одном подразделе
- [ ] **G.2 Нет «дыр» в развитии:** для каждой ступени (1-5) — есть подраздел, который её адресует
- [ ] **G.3 Bottleneck-узлы помечены:** подразделы, критичные для перехода между ступенями (gate Д→П, например), явно отмечены

### 🟡 H. Связность с другими руководствами

- [ ] **H.1 Cross-references корректны:** ссылки вида «см. Руководство 2, Раздел 3» ведут на существующие S
- [ ] **H.2 Дублирования нет:** концепции, введённые в другом руководстве, не переопределяются здесь — только используются
- [ ] **H.3 Точки сопряжения объяснены:** в местах, где этот гайд опирается на другой, дана 1-2 строки контекста (не заставлять читателя бросать чтение)

### 🟡 I. Эпилог и навигация

- [ ] **I.1 Эпилог есть:** руководство заканчивается явным итогом «что вы теперь умеете / видите иначе»
- [ ] **I.2 Связь со ступенями:** в эпилоге сказано, на какую ступень руководство выводит и куда дальше (следующее руководство / следующая программа)
- [ ] **I.3 Навигация по руководству:** README или оглавление содержит ясную карту разделов, рекомендованный порядок чтения

## Уровень 🟢: Пилот-проверка (живой человек на руководстве целиком)

> Не для каждого подраздела отдельно, а после прохождения руководства полностью (типично 2-4 недели по 1-2 подраздела в день).

### 🟢 J. Пилот-тест руководства (6 критериев)

≥3 пилотов проходят полностью, отвечают **да / нет / частично**:

- [ ] **J.1 Объект руководства понятен:** я могу одной фразой объяснить, о чём это руководство
- [ ] **J.2 Прогрессия чувствуется:** я заметил, что между началом и концом мой взгляд изменился
- [ ] **J.3 Нет «дыр»:** не было моментов «нужная мне ситуация не покрыта»
- [ ] **J.4 Нет дублирования:** не было моментов «это уже было в другом подразделе/руководстве»
- [ ] **J.5 Навигация работает:** я мог найти нужный подраздел через оглавление за <1 мин
- [ ] **J.6 Мировоззренческий сдвиг произошёл:** к концу я вижу свою практику/работу иначе, чем в начале

**Правило приёмки:** ≥3 пилота × 6/6 → `guide-approved`. <3 пилотов или <6/6 → `guide-rework` с привязкой к конкретным разделам.

## Алгоритм применения чек-листа

### Для агента (Claude Code):

```
1. Получить guide_id (например, "guide-1-system-self-development")
2. Уровень 🔴: запустить v4-lint guide-mode (A-D)
   - FAIL → остановиться, диагностика автору
   - PASS → перейти к 🟡
3. Уровень 🟡: делегировать субагенту Claude Opus
   - Передать в prompt: все S/SS файлы руководства + этот чек-лист §🟡 (E-I)
   - Получить отчёт PASS/FAIL с цитатами
   - FAIL → возврат автору с замечаниями
   - PASS → подраздел готов к пилот-тесту
4. Уровень 🟢: запустить пилот-тест (≥3 пилотов, асинхронно)
   - Сбор issue с label `guide-feedback`
   - Решение `guide-approved` / `guide-rework`
```

### Когда применять

| Триггер | Полный 🔴+🟡+🟢 | Только 🔴+🟡 | Только 🔴 |
|---------|-----------------|--------------|------------|
| Релиз нового руководства | ✅ | — | — |
| Ежеквартальный аудит | ✅ | — | — |
| Изменение Pack (FORM/METHOD) | — | ✅ | — |
| Hotfix typo во всём руководстве | — | — | ✅ |
