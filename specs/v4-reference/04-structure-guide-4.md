## Раздел 1. Ввод: системное мышление - это не систематичность

```yaml
section_id: PD.GUIDE.4.S1
title: "Ввод: системное мышление - это не систематичность"
order: 1
stage_relevant: [3, 4, 5]
cp_check: [cp.int, cp.wld]
bh_check: [bh.awr, bh.sys]
bottleneck_hint: "Если застрял - проверь cp.int: есть ли пример системного разбора из собственной жизни?"
```

### Ключевые понятия (граф → meta-meta)

| Понятие | Определение | Meta-meta связь (FPF) |
|---------|-------------|----------------------|
| Системное мышление | Мышление с понятиями трансдисциплин интеллект-стека: способность выделять системы, роли, границы, эмерджентность | Мышление → модель → представление |
| Систематичность | Регулярность, дисциплина, порядок в действиях | Поведение → паттерн → структура |
| Различение | Операция, фиксирующая границу между понятиями | Различение → основа мышления → единица анализа |
| Интеллект-стек | 16 трансдисциплин, прошивающих интеллект (FORM.085) | Знание → структура → карта |
| Инструмент мышления | Метод или модель, которую можно применять к любому объекту | Инструмент → посредник → расширение способности |
| Мировоззрение | Машинный код: инвариантные различения, через которые интерпретируется мир | Программа → рамка → фильтр |

### Содержание (план ~3000 слов)

#### Теория / различение
- Системное мышление ≠ систематичность. Систематичный человек соблюдает распорядок, но не обязательно видит системы. Системный мыслитель видит структуру, даже если она беспорядочна (FORM.085 §1)
- Различение как атом мышления. Пока не проведена граница между двумя понятиями, мышление работает на смеси. Примеры: роль ≠ исполнитель, система ≠ сумма частей, метод ≠ инструмент
- Интеллект-стек: 16 трансдисциплин от онтологии до теории решения изобретательских задач. Системное мышление - одна из них, но опирается на все остальные
- Инструмент vs мировоззрение: системное мышление можно использовать как инструмент («применю системный разбор к этой задаче»), но при регулярном применении оно становится мировоззрением - способом видеть мир по умолчанию
- Зачем это пилоту: проблема → задача. Системный мыслитель не решает «проблему» (неопределенность), а переводит её в задачу (определенность)

#### Метод / практика
- Тест «систематичность или системность?»: 3 ситуации из жизни пилота. Какие из них требуют порядка, а какие - разбора?
- Практика проведения одного различения: выбрать пару понятий, которые пилот смешивает, и четко отделить их по 3 признакам

#### Через IWE
- Pack как инструмент системного мышления: структура FORM/METHOD/CAT/SOTA - это системная карта знания
- Агент-Интеллектуал: переводит неопределенность в задачи через системный разбор
- Drift-detection: когда понятия в Pack дрейфуют, системное мышление ловит несоответствие

#### Проверка
- cp.int ≥ 2: пилот может назвать 2 различения из Pack и провести границу между ними
- bh.awr ≥ 2: замечает, когда смешивает понятия в разговоре или мышлении

### Практика (40 мин)
- Выбрать одну «проблему» из текущей жизни и перевести её в задачу через системный разбор (4 вопроса из FORM.086)
- Зафиксировать в Pack: какое различение было проведено, какая граница установлена

## Раздел 2. Эволюция системного подхода: 1.0 → 2.0 → 3.0

```yaml
section_id: PD.GUIDE.4.S2
title: "Эволюция системного подхода: 1.0 → 2.0 → 3.0"
order: 2
stage_relevant: [3, 4, 5]
cp_check: [cp.int, cp.wld]
bh_check: [bh.awr, bh.sys]
bottleneck_hint: "Если застрял - проверь cp.wld: понимает ли пилот, что система существует только при наличии интересов?"
```

### Ключевые понятия (граф → meta-meta)

| Понятие | Определение | Meta-meta связь (FPF) |
|---------|-------------|----------------------|
| Системный подход 1.0 | Классический: целостность, эмерджентность, вложенность. Объект - система как объект физического мира | Объект → вещь → данность |
| Системный подход 2.0 | Интересный: нет интересов - нет системы. Система выделяется вниманием заинтересованных лиц | Интерес → направленность → выделение |
| Системный подход 3.0 | Безмасштабный: деантропоморфность, многоуровневая эволюция, системы без наблюдателя | Масштаб → уровень → охват |
| Эмерджентность | Появление новых свойств на системном уровне, не предсказуемых из свойств частей | Целое → свойство → уровень |
| Вложенность | Система состоит из подсистем и входит в надсистемы | Иерархия → структура → уровень |
| Деантропоморфность | Отказ от проецирования человеческих свойств на нечеловеческие системы | Объективность → независимость от наблюдателя |

### Содержание (план ~4000 слов)

#### Теория / различение
- Системный подход 1.0: три кита - целостность (система ≠ сумма частей), эмерджентность (новые свойства на уровне целого), вложенность (подсистемы и надсистемы). Пример: атом → молекула → клетка → организм. Каждый уровень дает новые свойства (FORM.086 §2)
- Системный подход 2.0: революция - система не существует без заинтересованных лиц. Один и тот же объект может быть системой для одного наблюдателя и хаосом для другого. Успешная система учитывает интересы всех заинтересованных лиц, не удовлетворяет всех - учитывает
- Таблица 3×3: 3 области интересов ролей (создатель, оператор, пользователь) × 3 области интересов систем (целевая, система создания, системы в окружении). Девять ячеек - девять точек зрения на одну систему
- Системный подход 3.0: безмасштабность - системные законы работают на любом уровне (атом, организация, галактика). Деантропоморфность: не надо думать, что организация «хочет» что-то. Системы не хотят - они функционируют
- Эволюция взгляда: от «система - объект» к «система - интерес» к «система - закономерность». Каждый следующий уровень не отменяет предыдущий, а вкладывает его

#### Метод / практика
- Диагностика: на каком подходе мыслит пилот по умолчанию? Тест из 5 ситуаций
- Практика «смены линзы»: взять одну систему из жизни пилота и рассмотреть её через 1.0, 2.0, 3.0

#### Через IWE
- IWE как система 2.0: есть создатель (пилот), оператор (агенты), пользователь (пилот же, но в другой роли). Интересы каждой роли разные
- Pack отражает 3.0: безмасштабные структуры FORM/METHOD работают и для личности, и для организации
- Агент-Стратег: работает на уровне 2.0 - учитывает интересы всех ролей пилота

#### Проверка
- cp.int ≥ 3: пилот может объяснить разницу между 1.0, 2.0, 3.0 на примере
- bh.sys ≥ 2: пилот видит системы вокруг себя, а не только объекты

### Практика (40 мин)
- Выбрать одну систему из текущей жизни (работа, проект, отношения). Рассмотреть её через таблицу 3×3: 9 ячеек, краткие записи
- Зафиксировать в Pack: какая ячейка была неочевидной? Какие интересы ранее не учитывались?

## Раздел 3. Различения: роль, граница, эмерджентность, функция

```yaml
section_id: PD.GUIDE.4.S3
title: "Различения: роль, граница, эмерджентность, функция"
order: 3
stage_relevant: [3, 4, 5]
cp_check: [cp.int, cp.skl]
bh_check: [bh.awr, bh.met]
bottleneck_hint: "Если застрял - проверь cp.skl: может ли пилот назвать функцию системы, а не только её состав?"
```

### Ключевые понятия (граф → meta-meta)

| Понятие | Определение | Meta-meta связь (FPF) |
|---------|-------------|----------------------|
| Роль (системный смысл) | Функция, которую выполняет элемент в системе. Не должность, не личность | Функция → место в системе → назначение |
| Граница системы | Линия, отделяющая систему от окружения. Проводится вниманием, не дана природой | Граница → различение → единица анализа |
| Эмерджентность | Свойства системного уровня, не сводимые к свойствам частей. Предсказуемы через моделирование | Целое → новое качество → уровень |
| Функция | Желаемое действие системы. Отвечает на вопрос «зачем?» | Цель → направление → смысл |
| Целевая система | Система, для которой создается решение | Объект → цель → фокус |
| Система создания | Система, которая создает целевую систему | Производитель → источник → причина |
| Цепочка создания | Последовательность систем создания от идеи до эксплуатации | Процесс → цепь → этап |
| Оргзвено / оргроль / сервис | Три уровня организации как системы: структура, функция, предоставляемая ценность | Организация → система → сервис |

### Содержание (план ~4000 слов)

#### Теория / различение
- Роль ≠ должность. Должность - организационная единица, роль - функция. Один человек может выполнять несколько ролей, одна роль - несколькими людьми. Пример: в проекте «роль аналитика» может выполнять разработчик, менеджер или внешний консультант (FORM.086 §3)
- Граница системы - самое важное различение. Где система заканчивается и начинается окружение? Граница проводится наблюдателем и зависит от интересов. Пример: для биолога организм - система, для эколога - часть экосистемы
- Эмерджентность: вода мокрая, но атомы водорода и кислорода по отдельности не мокрые. Мокрость - эмерджентное свойство. В социальных системах: команда из 5 людей может быть продуктивной или токсичной - это не свойство отдельных людей
- Функция vs состав. Состав - «из чего», функция - «зачем». Системный мыслитель начинает с функции, не с состава. Пример: молоток - это функция «вбивать гвозди», а не форма «ручка + боек»
- Целевая система, система создания, надсистема. Цепочка создания: идея → проектирование → производство → эксплуатация. На каждом этапе своя система создания
- Оргзвено (структурная единица) → оргроль (функция) → сервис (ценность для получателя). Организация работает, когда сервис доставлен, а не когда оргзвено отчиталось

#### Метод / практика
- Практика «граница внимания»: выбрать объект, провести 3 разные границы (для разных интересов), записать, что входит в систему и что - в окружение
- Практика «ролевой анализ»: разобрать одну систему на роли, не на людей

#### Через IWE
- Роли в IWE: Пилот (функция мышления), Машина (функция исполнения), Агенты (функции усиления). Каждая роль - не личность, а функция
- Pack: структура отражает цепочку создания знания - от форм к методам к SOTA
- Агент-Диагност: функция - выявлять bottleneck, не решать задачи

#### Проверка
- cp.int ≥ 3: пилот различает роль и должность, функцию и состав
- bh.met ≥ 2: пилот применяет различение «функция vs состав» в реальных задачах

### Практика (40 мин)
- Системный разбор одной организации или проекта из жизни пилота: роли (не люди), границы (3 варианта), функция (не состав)
- Артефакт: системная карта на бумаге или в Pack. Минимум: 3 роли, 1 граница, 1 функция

## Раздел 4. Системный разбор: 4 вопроса и системная карта

```yaml
section_id: PD.GUIDE.4.S4
title: "Системный разбор: 4 вопроса и системная карта"
order: 4
stage_relevant: [3, 4, 5]
cp_check: [cp.int, cp.cre]
bh_check: [bh.awr, bh.met]
bottleneck_hint: "Если застрял - проверь bh.met: есть ли применение системного разбора к реальной задаче, а не только теория?"
```

### Ключевые понятия (граф → meta-meta)

| Понятие | Определение | Meta-meta связь (FPF) |
|---------|-------------|----------------------|
| Системный разбор | Метод анализа через 4 вопроса: зачем, что, как, когда (FORM.086 §4) | Метод → алгоритм → способ |
| Системная карта | Артефакт разбора: схема системы с границами, ролями, потоками, ограничениями | Модель → представление → карта |
| Модель / мета-модель / мульти-модель / мега-модель | Уровни описания: одна модель → модель моделей → множество моделей → единая картина | Описание → уровень → охват |
| Чёрный ящик | Внешнее описание системы: входы, выходы, функция. Не раскрываем устройство | Интерфейс → поведение → наблюдение |
| Прозрачный ящик | Внутреннее описание: состав, процессы, механизмы | Структура → устройство → анализ |
| Интерфейс | Правила взаимодействия системы с окружением | Граница → связь → обмен |
| Характеристика / состояние / показатель | Свойства системы: характеристика - потенциал, состояние - момент, показатель - измерение | Свойство → качество → метрика |
| Процесс | Взаимодействие систем во времени. Процесс ≠ система | Динамика → изменение → поток |
| Проект | Временная деятельность по созданию уникальной системы | Задача → ограниченное время → уникальность |
| Мета-системный переход | Переход на более высокий системный уровень для решения проблемы текущего уровня | Уровень → охват → решение |

### Содержание (план ~4500 слов)

#### Теория / различение
- 4 вопроса системного разбора (FORM.086 §4):
  1. Зачем? - функция, цель, интересы заинтересованных лиц
  2. Что? - границы системы, состав, роли, надсистема и подсистемы
  3. Как? - процессы, методы, механизмы, ограничения
  4. Когда? - временная структура, стадии, этапы, итерации, инкременты
- Системная карта - это не схема красивых квадратиков. Это артефакт, где каждый элемент отвечает на один из 4 вопросов. Плохая карта: красивая, но не отвечает на «зачем». Хорошая карта: может быть уродливой, но все элементы обоснованы
- Чёрный ящик vs прозрачный ящик. На этапе «зачем» система - чёрный ящик: важно, что она делает, не как устроена. На этапе «как» - прозрачный: важно устройство. Умение переключать уровень детализации - ключевой навык
- Уровни моделей: модель (одно описание) → мета-модель (описание того, как строить модели) → мульти-модель (несколько моделей одного объекта) → мега-модель (интегрированная картина). Пилоту нужна мульти-модель: несколько точек зрения на одну систему
- Процесс ≠ система. Система - объект, процесс - взаимодействие систем во времени. Проект - временная деятельность по созданию системы. Мета-системный переход: когда проблема не решается на текущем уровне, поднимаемся на уровень выше
- Стадия / этап / итерация / инкремент: стадия - фаза жизненного цикла, этап - часть стадии, итерация - цикл обратной связи, инкремент - приращение результата

#### Метод / практика
- Алгоритм построения системной карты: начать с «зачем», провести границу, выделить роли, найти потоки, обозначить ограничения
- Правило «5 элементов»: если на карте больше 5 элементов - проверить, все ли нужны. Избыточная детализация = маскировка непонимания

#### Через IWE
- Pack-структура как системная карта: FORM - «зачем», METHOD - «как», CAT - «что», SOTA - «когда (текущее состояние)"
- Агент-Интеллектуал: применяет системный разбор к задачам пилота. Пилот видит процесс и учится
- RCS-профиль: системная карта пилота - какие роли, какие потоки, какие ограничения

#### Проверка
- cp.int ≥ 3: пилот может провести системный разбор по 4 вопросам без подсказок
- cp.cre ≥ 2: пилот создает системные карты как рабочие продукты

### Практика (40 мин)
- Построить системную карту одной реальной задачи из жизни пилота по 4 вопросам. Артефакт: текстовое описание или схема
- Проверка карты: каждый элемент отвечает на один из 4 вопросов? Есть ли граница? Есть ли роли? Зафиксировать в Pack

## Раздел 5. Себя и IWE как системы: Болид, организм, потоки

```yaml
section_id: PD.GUIDE.4.S5
title: "Себя и IWE как системы: Болид, организм, потоки"
order: 5
stage_relevant: [3, 4, 5]
cp_check: [cp.wld, cp.iwe]
bh_check: [bh.sys, bh.awr]
bottleneck_hint: "Если застрял - проверь cp.iwe: понимает ли пилот Machine как систему с границами и ролями?"
```

### Ключевые понятия (граф → meta-meta)

| Понятие | Определение | Meta-meta связь (FPF) |
|---------|-------------|----------------------|
| Болид | Пилот + Машина, неделимая система (FORM.089 §1) | Система → целостность → эмерджентность |
| Организм | Биологическая система пилота: тело, мозг, нервная система | Подсистема → база → платформа |
| Личность | Система психических свойств: мировоззрение, роли, отношения | Подсистема → программа → интерпретатор |
| Мастерство | Накопленная способность производить результаты | Подсистема → компетенция → капитал |
| Система «я» | Организм + личность + мастерство. Три подсистемы, дающие эмерджентность: живой человек ≠ тело + мысли + навыки | Целое → эмерджентность → личность |
| IWE как система | Репозитории, среда, агенты, Pack - система создания для пилота | Инструмент → среда → расширение |
| Три потока личной жизни | Рабочие продукты, информация (обучение + память), энергия (PD.METHOD.023) | Поток → движение → ресурс |

### Содержание (план ~3500 слов)

#### Теория / различение
- Болид-онтология: человек не отделен от инструментов. Но здесь фокус не на философии, а на системном разборе. Болид - система, у которой есть целевая функция (созидание), система создания (пилот + машина), надсистема (среда, сообщество) (FORM.089 §1)
- Система «я» - три подсистемы: организм (биология), личность (психика), мастерство (компетенции). Каждая подсистема имеет свои ограничения, свои показатели, свои потоки. Проблема в одной подсистеме проявляется в другой: усталость организма = когнитивное ограничение
- IWE как система создания для пилота. Целевая система - рабочие продукты пилота. Система создания - IWE (репозитории, агенты, методы). Надсистема - среда: сообщество, рынок, культура
- Три потока личной жизни (PD.METHOD.023): внешние рабочие продукты (что создается), внутренняя информация (что изучается, что запоминается), внутренняя энергия (восстановление, мотивация, состояние). Блокировка одного потока останавливает остальные
- Саморазвитие = изменение системы «я». Не накопление знаний, а изменение структуры системы: новые роли, новые границы, новые потоки

#### Метод / практика
- Практика «разбор себя как системы»: 4 вопроса применить к системе «я»
- Карта «три потока»: что входит в каждый поток, где bottleneck, где переполнение

#### Через IWE
- IWE отражает систему «я» в Pack: структура репозиториев = структура интересов, ролей, проектов
- WakaTime и Git - показатели состояния потоков: сколько Touch Time, сколько Wait Time
- Агент-Диагност: алерт, когда один из трёх потоков пустует более 5 дней

#### Проверка
- cp.wld ≥ 3: пилот видит себя как систему из 3 подсистем, не как «я просто такой»
- cp.iwe ≥ 2: пилот может описать Machine как систему с границами и функциями

### Практика (40 мин)
- Системный разбор «я»: 4 вопроса (зачем, что, как, когда) применительно к себе как системе. Артефакт: текстовая системная карта
- Карта трёх потоков: что сейчас в каждом потоке, где bottleneck. Зафиксировать в Pack

## Раздел 6. Tame Flow: потоки, ограничения и метрики

```yaml
section_id: PD.GUIDE.4.S6
title: "Tame Flow: потоки, ограничения и метрики"
order: 6
stage_relevant: [3, 4, 5]
cp_check: [cp.int, cp.skl]
bh_check: [bh.met, bh.inv]
bottleneck_hint: "Если застрял - проверь bh.inv: есть ли данные о времени (WakaTime, трекинг), чтобы считать Flow Efficiency?"
```

### Ключевые понятия (граф → meta-meta)

| Понятие | Определение | Meta-meta связь (FPF) |
|---------|-------------|----------------------|
| Tame Flow | Метод операционного менеджмента (Стив Тендон): управление потоками работ через ограничения, метрики и предварительную подготовку | Метод → управление потоком → операционка |
| Закон Литтла | WIP = Flow Time × Throughput. Фундаментальное уравнение операционного менеджмента | Уравнение → инвариант → закон |
| Flow Time / Lead Time | Время от начала работы до готового результата | Длительность → период → метрика |
| Touch Time | Время, когда работа реально выполняется | Активное время → эффективность → качество |
| Wait Time | Время ожидания: задача готова, но ресурс занят | Пассивное время → потеря → неэффективность |
| Flow Efficiency | Touch Time / Flow Time. Бенчмарки: <15% (без управления), 15–40% (типично), 60%+ (гиперпродуктивность) | Эффективность → отношение → показатель |
| WIP | Незавершённая работа. Ограничение WIP = основной рычаг управления скоростью | Нагрузка → объём → инвентарь |
| Full Kitting | Предварительная подготовка всего необходимого до начала работы | Подготовка → условие → чистый Touch Time |
| Five Focusing Steps | Identify → Exploit → Subordinate → Elevate → Repeat. Алгоритм управления bottleneck (Голдратт) | Алгоритм → управление ограничением → цикл |
| Constraint / Bottleneck | Фаза/этап/ресурс, определяющий общую пропускную способность системы. Не «проблема», а управляемый параметр | Ограничение → регулятор потока → точка воздействия |
| Трихотомия ограничений | Work Flow (поперёк pipeline), Work Process (внутри этапа), Work Execution (в моменте) | Уровень → масштаб → локализация |
| Политика vs Ресурс | Дискриминатор (Шрагенхайм): bottleneck стабилен → ресурс; bottleneck мигрирует → политика | Причина → тип → стратегия |
| Cognitive freeze | Когнитивное ограничение: рост открытых вопросов без решений, прокрастинация | Mental model → ограничение → узкое место |
| Throughput Accounting | T/I/OE: Throughput / Inventory / Operating Expense | Бухгалтерия → поток → метрика |

### Содержание (план ~5000 слов)

#### Теория / различение
- Tame Flow - применение системного подхода к управлению работой. Работа - это поток задач через систему «я» + Machine. Поток можно измерять, ограничивать, ускорять (WP-313, D-004)
- Закон Литтла: WIP = Flow Time × Throughput. Это не «совет», это математический инвариант. Если WIP растет, а Throughput не меняется - Flow Time растет пропорционально. Больше открытых задач = дольше каждая задача
- Flow Time = Touch Time + Wait Time. В knowledge work Flow Efficiency часто <15%: 8 часов «работы» = 1 час реальной работы + 7 часов ожидания, переключений, поиска
- Flow Efficiency как показатель здоровья системы. <15% - система хаотична, 15–40% - управляема, 60%+ - гиперпродуктивность (редко достижима, но возможна при Full Kitting + ограничение WIP)
- Full Kitting: перед началом работы подготовить всё необходимое - материалы, инструменты, метод, приоритет, фокус. Цель: сделать сессию чистым Touch Time. Без Full Kitting 30–50% сессии уходит на подготовку в процессе
- Five Focusing Steps: Identify (найти bottleneck) → Exploit (использовать на полную) → Subordinate (подчинить остальное bottleneck) → Elevate (усилить bottleneck) → Repeat (вернуться к шагу 1, bottleneck мигрирует)
- Bottleneck - не проблема, а управляемый параметр. В системном подходе bottleneck - это точка воздействия. Изменение bottleneck дает максимальный эффект при минимальных затратах
- Трихотомия ограничений: Work Flow - что тормозит весь pipeline (самая длинная загрузка), Work Process - что тормозит внутри этапа (самое длинное среднее потоковое время), Work Execution - что тормозит прямо сейчас (динамическое)
- Политика vs Ресурс: если bottleneck стабилен и усиление ресурса помогает - это ресурсное ограничение. Если bottleneck мигрирует и усиление не помогает - это политика (правила, которые создают ограничение)
- Cognitive freeze: mental models = ultimate constraint в knowledge work. Когда много открытых вопросов и нет решений - работа стопорится. Лечение: системный разбор, EC, NBR
- Throughput Accounting: в личной работе Throughput = завершенные рабочие продукты, Inventory = незавершенные задачи/PR, Operating Expense = время на поддержание. Рост Inventory без роста Throughput = прокрастинация в маскировке «занятости"

#### Метод / практика
- Подсчёт Flow Efficiency за неделю: Touch Time / Flow Time. Источники данных - WakaTime, календарь, трекинг
- Практика ограничения WIP: установить лимит открытых задач (например, 3) и не брать новые, пока старые не закрыты
- Full Kitting чек-лист: 6 пунктов перед каждой сессией

#### Через IWE
- WakaTime - Touch Time трекинг. Git - Throughput (коммиты, PR). Календарь - Flow Time (от начала до конца задачи)
- Агент-Навигатор: алерт при росте WIP (открытых задач/PR > лимита)
- Pack: шаблон Full Kitting для типовых сессий (кодинг, письмо, планирование)

#### Проверка
- cp.int ≥ 3: пилот понимает Закон Литтла и может объяснить связь WIP и Flow Time
- bh.inv ≥ 2: пилот трекает время и может посчитать Flow Efficiency за неделю

### Практика (40 мин)
- Подсчитать Flow Efficiency за прошлую неделю: Touch Time (WakaTime или оценка) / Flow Time (от постановки задачи до результата)
- Найти bottleneck в текущей работе: какой из трёх типов (Work Flow / Work Process / Work Execution)?
- Составить Full Kitting чек-лист для одного типа сессий. Зафиксировать в Pack

## Раздел 7. Инструменты решения: EC, NBR и когнитивные ограничения

```yaml
section_id: PD.GUIDE.4.S7
title: "Инструменты решения: EC, NBR и когнитивные ограничения"
order: 7
stage_relevant: [3, 4, 5]
cp_check: [cp.int, cp.cre]
bh_check: [bh.met, bh.agn]
bottleneck_hint: "Если застрял - проверь bh.agn: есть ли пример применения EC или NBR к реальному конфликту?"
```

### Ключевые понятия (граф → meta-meta)

| Понятие | Определение | Meta-meta связь (FPF) |
|---------|-------------|----------------------|
| Evaporating Cloud (EC) | Инструмент разрешения конфликтов: 5 элементов (Common Goal, Need A, Need B, Want D, Want D'), скрытые допущения, injection | Метод → разрешение конфликта → синтез |
| Negative Branch Reservation (NBR) | Проверка рисков: 3 негативных ветви после injection + обрезка каждой | Метод → управление риском → защита |
| Injection | Действие или условие, которое разрушает конфликт EC, не жертвуя ни одной need | Решение → прорыв → выход |
| Скрытое допущение | Предположение, лежащее в основе конфликта. Часто ложное или необязательное | Убеждение → фильтр → ограничение |
| Когнитивное ограничение (cognitive freeze) | Mental models как ultimate constraint: когда мышление блокирует действие | Модель → ограничение → bottleneck |
| Конфликт (системный) | Столкновение интересов или needs, а не людей. Конфликт решается на уровне допущений, не компромиссов | Напряжение → разрыв → энергия |
| Компромисс | Ущерб одной need ради другой. EC ищет injection, а не компромисс | Сделка → потеря → компромисс |
| Обрезка ветви (NBR) | Действие, предотвращающее негативное следствие injection | Превентивное действие → защита → контроль |

### Содержание (план ~4500 слов)

#### Теория / различение
- EC (Evaporating Cloud): 5 элементов - Common Goal (общая цель), Need A (потребность стороны A), Need B (потребность стороны B), Want D (действие A), Want D' (действие B). D и D' конфликтуют, но конфликт не в действиях, а в скрытых допущениях (WP-313 §7)
- Пример системного конфликта: «Нужно работать больше» vs «Нужно отдыхать больше». Common Goal: продуктивность и здоровье. Need A: результат. Need B: восстановление. Want D: 12 часов работы. Want D': 8 часов сна. Скрытое допущение: «больше часов работы = больше результата». Injection: «сократить WIP и применить Full Kitting → тот же результат за 6 часов, оставив 8 на сон»
- NBR (Negative Branch Reservation): любое injection имеет негативные следствия. НBR требует найти 3 негативные ветви и обрезать каждую до применения injection. Пример: injection «сократить WIP» → ветвь «упущенные возможности» → обрезка: «WIP-лимит только на текущие задачи, новые попадают в backlog"
- Конфликт ≠ спор. Системный конфликт - это структура, не эмоция. EC работает с интересами (needs), не с позициями (wants). Позиции конфликтуют, интересы - редко
- Cognitive freeze: когда открытых вопросов больше, чем закрытых, работа останавливается. Mental models (убеждения о том, как работает мир) - ultimate constraint. Примеры: «я должен всё знать перед началом», «если не сделаю сам - не будет хорошо». EC и NBR работают с mental models напрямую
- Компромисс vs injection. Компромисс = обе стороны недовольны. Injection = обе needs удовлетворены, конфликт исчезает (evaporates)

#### Метод / практика
- Алгоритм EC: записать Common Goal → 2 Needs → 2 Wants → найти допущения → сформулировать injection → проверить NBR
- Алгоритм NBR: взять injection → найти 3 негативных следствия → для каждого: обрезка (действие, предотвращающее следствие)
- Практика «обрезка»: какое действие сделает негативное следствие невозможным?

#### Через IWE
- Pack: шаблон EC и NBR как методы (METHOD.XXX). Заполнение EC в Pack → артефакт для обсуждения с Агентом
- Агент-Стратег: помогает провести EC, находит скрытые допущения, которые пилот не видит
- Агент-Диагност: алерт при cognitive freeze (рост открытых вопросов без решений > 3 дней)

#### Проверка
- cp.int ≥ 3: пилот может провести EC для одного реального конфликта
- cp.cre ≥ 2: пилот применяет NBR перед внедрением решений
- bh.agn ≥ 2: пилот инициирует изменения на основе системного разбора, не ждёт «лучшего времени»

### Практика (40 мин)
- Провести EC для одного реального конфликта из жизни пилота. Артефакт: заполненная схема EC в Pack или на бумаге
- Для найденного injection - провести NBR: 3 негативных ветви + обрезка каждой
- Зафиксировать injection и обрезки в Pack как решение

## Что дальше: связь с программой «Рабочее развитие»

Руководство 4 даёт мыслительный инструментарий, который переходит в программу «Рабочее развитие» следующим образом:

| Метод / навык из руководства 4 | Переход в «Рабочее развитие» |
|-------------------------------|------------------------------|
| Системный разбор (4 вопроса) | Базовый инструмент для любого WP: задача ставится через разбор «зачем, что, как, когда» |
| Системная карта | Артефакт приёмки для WP-уровня задач: «покажи карту системы, которую создаешь» |
| Tame Flow (Flow Efficiency, WIP, Full Kitting) | Операционная база личной работы: трекинг, ограничение WIP, подготовка сессий |
| Five Focusing Steps | Алгоритм еженедельного планирования: найти bottleneck недели, подчинить остальное |
| EC и NBR | Инструмент разрешения личных и рабочих конфликтов, проверки рисков решений |
| Три потока личной жизни | Фрейм для баланса: рабочие продукты + обучение + энергия. Дисбаланс = алерт |
| Различение роль / должность / функция | База для ролевого планирования и работы в командах |
| Таблица 3×3 | Инструмент учёта интересов в проектах с несколькими заинтересованными лицами |

Пилот, прошедший руководство 4, может:
- Ставить задачи через системный разбор (не «надо сделать», а «зачем, что, как, когда")
- Видеть bottleneck в собственной работе и действовать через Five Focusing Steps
- Разрешать конфликты через EC, а не компромисс
- Проверять риски через NBR до принятия решений
- Управлять потоками работы через ограничение WIP и Full Kitting
