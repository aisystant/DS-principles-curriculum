# Чек-лист проверки подраздела (v1)

> **Применяется к:** одному подразделу (SS, типично 0.5-1.5K слов = дневная доза).
>
> **Иерархия чек-листов:** руководство → раздел → подраздел (этот). Это базовый уровень — всегда запускается первым. Index: [CHECKLISTS-README.md](CHECKLISTS-README.md).
>
> **Назначение:** агент Claude Code применяет этот чек-лист к готовому подразделу перед merge в staging.
>
> **Уровни:**
> - 🔴 **Машина** — `v4-lint` или скрипт. Exit 0 = PASS. Не требует понимания контекста.
> - 🟡 **Агент** — требует Claude (или эквивалент). Проверяет семантику, связность, тональность.
> - 🟢 **Пилот** — требует живого человека. Проверяет восприятие, аналогию, сдвиг.
>
> **Правило:** 🔴 должен быть 100% PASS перед 🟡. 🟡 должен быть 100% PASS перед 🟢.
> Hotfix-исключение: `[hotfix]` в коммите = только 🔴, без 🟡🟢.

## Уровень 🔴: Машинная проверка (v4-lint)

> Запускается через CLI. Exit 0 = PASS. Диагностика в stderr.

### 🔴 A. Структурная целостность

```bash
# Проверить ВСЕ подразделы в директории
python3 tools/v4-lint.py structure specs/v4-reference/
```

- [ ] **A.1 Нет пропусков SS:** каждый раздел имеет SS1–SS7 или `optional: true`
- [ ] **A.2 Численная сортировка:** S1, S2, ..., S10 (не S1, S10, S2)
- [ ] **A.3 Омонимы разрешены:** одно слово с разным parent U.* — имеет квалификатор
- [ ] **A.4 Нет unknown маркеров:** строка без `:` или неизвестный маркер → FAIL
- [ ] **A.5 Формат «Понятия:»:** имя не начинается с `(`
- [ ] **A.6 Тройка идентификации:** каждое «вводится» имеет имя + parent U.*
- [ ] **A.7 Онтологическая бережливость (STRUCT-PARSIMONY):** ≤3 «вводится» на подраздел (WARN на 4+, FAIL на 6+)
- [ ] **A.8 Evidence Graph (STRUCT-EVIDENCE):** каждое «вводится» имеет источник Pack `(PD.FORM/METHOD/CAT.NNN)` — **FAIL если нет** (Pack-sufficiency gate, см. PD.FORM.103 Этап 3.5 + WRITING-PIPELINE §1.5). Severity поднята с WARN до FAIL 17 мая 2026 (Ф0.14, выравнивание с блокирующим правилом Pack-first; миграция: проверить guide 1-4 на orphan-концепты до включения в CI)
- [ ] **A.9 Кейс в introduces:** нет имён собственных в «вводится» (Земмельвейс, Хохланд — FAIL)
- [ ] **A.10 Шифры Pack в `introduces`:** запрещены `PD.FORM.NNN`, `PD.METHOD.NNN`, `cp.*`, `bh.*` внутри значений `introduces` — FAIL
- [ ] **A.11 Формат `prerequisites`:** только `PD.GUIDE.N.SX.SSY`. `§X.YY` → FAIL

### 🔴 B. Контракт Портного (frontmatter)

```bash
python3 tools/v4-lint.py porter <файл_подраздела.md>
```

- [ ] **B.1 Поля присутствуют:** `subsection_id`, `title`, `mastery_node`, `stage_relevant`, `introduces`, `uses`, `prerequisites`, `can_do`, `cp_check`, `bh_check`
- [ ] **B.2 mastery_node валиден:** ∈ {мыслительное, саморазвитие, iwe}
- [ ] **B.3 Bounded Context (A.1.1):** `mastery_node: [iwe]` запрещён в guide 1/2 → FAIL
- [ ] **B.4 stage_relevant валиден:** подмножество [1,2,3,4,5]
- [ ] **B.5 can_do формат:** каждый элемент начинается с «Могу»
- [ ] **B.6 introduces канонично:** только имена, без `U.*` префикса
- [ ] **B.7 prerequisites существуют:** каждый `prerequisites` ссылается на реальный `subsection_id`
- [ ] **B.8 subsection_id формат:** `PD.GUIDE.N.SX.SSY` — уникален в пределах 4 руководств
- [ ] **B.9 Обязательные meta-поля:** `time_reading`, `time_practice`, `word_count_target`, `status`, `wp`, `format_version` — FAIL если отсутствует в main-подразделе; WARN в auxiliary
- [ ] **B.10 Pack-refs не пустой:** если `introduces` не пуст — `pack_refs` обязан содержать ≥1 запись с `concept:` и `ref:` (или `pack:`), соответствующую каждому понятию из `introduces`. Пустой `pack_refs: []` при непустом `introduces` → **FAIL**. Исключения (whitelist) вводятся только при ≥3 файлах с явным обоснованием статуса «без Pack-источника»; до этого момента все существующие `pack_refs: []` трактуются как технический долг, а не легитимный паттерн.

### 🔴 C. Кросс-руководная согласованность

```bash
python3 tools/v4-lint.py cross-guide specs/v4-reference/
```

- [ ] **C.1 Одно понятие = одно определение:** понятие в `introduces` встречается ровно 1 раз
- [ ] **C.2 Нет orphan-references:** `uses` ссылается на существующее `introduces` (WARN если нет)
- [ ] **C.3 Нет дублей узлов:** `v4-lint graph build` → нет WARN «уже введено»

### 🔴 D. Pack-drift

```bash
python3 tools/v4-lint.py pack-drift specs/v4-reference/ --pack <путь_к_PD.FORM.089>
```

- [ ] **D.1 Все cp.* / bh.* актуальны:** неизвестный индекс → FAIL с location
- [ ] **D.2 Pack source указан:** `--pack` обязателен, нет hardcoded fallback

### 🔴 E. Граф понятий (консистентность)

```bash
python3 tools/v4-lint.py graph build specs/v4-reference/ --out-json graph.json
```

- [ ] **E.1 Граф собирается без ошибок:** exit 0
- [ ] **E.2 Нет циклов:** ациклический граф (проверяется визуально или скриптом)
- [ ] **E.3 Все introduces имеют узел:** каждое понятие из `introduces` — в графе

### 🔴 F. Git-целостность

- [ ] **F.1 `git status` чистый:** нет незакоммиченных изменений в `specs/v4-reference/`
- [ ] **F.2 Коммит подписан:** сообщение коммита содержит `WP-XXX` или `[v4]`
- [ ] **F.3 Файлы в правильном репо:** structure → `DS-principles-curriculum`, текст → `aisystant/docs`

### 🔴 F.4 Формат степеней мастерства (main-подразделы)

- [ ] **F.4.1 Таблица:** 4 степени оформлены таблицей «Степень / Что происходит / Критерий перехода» — FAIL на нумерованном списке
- [ ] **F.4.2 Проверка себя:** 3 индикатора (понимание / поведение / застревание) — WARN на простом списке вопросов
- [ ] **F.4.3 Нумерованный список степеней (v4.1):** блок «Степени мастерства» использует нумерованный список `1. / 2. / 3. / 4.` (не таблицу `|`). Регекс: `^1\.` после заголовка степеней. Таблица → **FAIL** как legacy-формат.

## Auxiliary-подразделы (.08–.11)

> **Правило:** `.08-concepts`, `.09-exercises`, `.10-review-questions`, `.11-section-conclusions` — сводные/практические страницы, не вводящие понятий. Полная v4-структура (13 блоков) к ним не применяется.

**Минимальный frontmatter:**
```yaml
---
subsection_id: PD.GUIDE.N.SX.SSY
title: "..."
order: N
guide: N
section: SN
format_version: 4.1-aux
---
```

**Что проверяется:**
- 🔴 `subsection_id`, `title`, `order` — присутствуют
- 🟡 Нет `introduces` / `uses` (это не подразделы-понятия)
- 🟡 Содержание соответствует типу (concepts = сводка понятий, exercises = практики, review = вопросы, conclusions = выводы раздела)

## Уровень 🟡: Агент-проверка (Claude / субагент)

> Агент читает текст и применяет семантические критерии. Требует context isolation: агент видит только результат, не процесс создания.
>
> **Метод:** Claude Code с файлом подраздела + этим чек-листом. Или `claude-code --file <подраздел.md>` с инструкцией «проверь по CHECKLIST-subsection-v1.md §🟡».

### 🟡 G. Границы понятий (A.6 Boundary Discipline)

- [ ] **G.1 Нет смешения:** «X включает Y» только если X ≠ Y и граница чёткая
- [ ] **G.2 Нет размытых определений:** каждое «вводится» имеет однозначный смысл в контексте руководства
- [ ] **G.3 Нет tautology:** определение не повторяет имя понятия другими словами

### 🟡 H. Нарративная цепочка

- [ ] **H.1 Мем → метод → мировоззрение:** в подразделе прослеживается цепочка:
  - Мем = знакомый образ/аналогия (понятно с первого предложения)
  - Метод = как делать (практика, алгоритм, шаги)
  - Мировоззрение = как смотреть на мир (связь с надсистемой)
- [ ] **H.2 Каждый абзац несёт одну новую мысль:** нет абзацев-повторений
- [ ] **H.3 Thinking Through Writing:** текст написан как процесс мышления, не как справочник
- [ ] **H.4 Блок «Что дальше»:** присутствует в конце подраздела, содержит отсылку к программе «Рабочее развитие» или старшим ступеням — FAIL если отсутствует

### 🟡 I. Дуга по ступени (тональность)

- [ ] **I.1 Ст. 1-2 (конкретика):**
  - Первый шаг описан конкретно (можно начать сегодня)
  - Нет абстракций «цивилизация», «эволюция", «будущее человечества»
  - Примеры из личного опыта или близкого окружения
- [ ] **I.2 Ст. 3-5 (горизонт):**
  - Связь с надсистемой обязательна (куда это ведёт, зачем миру)
  - Абстракции оправданы — они раскрывают, а не заменяют практику
  - Есть «дверь в бесконечность» — путь не заканчивается на подразделе

### 🟡 J. Практика и время

- [ ] **J.1 Практика описана:** есть конкретное задание с длительностью (15/25/40 мин)
- [ ] **J.2 Время реалистично:** практика выполнима заявленное время (не 5-минутная задача на 40 мин)
- [ ] **J.3 Результат практики измерим:** «что получится» описано конкретно
- [ ] **J.4 Детализация практики:** практика содержит ≥3 конкретных шага с длительностью или порядком — WARN если 1 абзац без пунктов

### 🟡 K. Аналогия и метафора

- [ ] **K.1 Аналогия есть:** подраздел использует ≥1 метафору/аналогию
- [ ] **K.2 Аналогия работает:** метафора создаёт новое понимание, не просто иллюстрирует
- [ ] **K.3 Нет смешения уровней:** аналогия из той же области (не смешивать биологию и IT без явного перехода)
- [ ] **K.4 Аналогии вне сигнатуры:** кейсы/аналогии присутствуют в тексте, но отсутствуют в `introduces` и в «Сигнатуре понятий» — FAIL если обнаружены в сигнатуре

### 🟡 L. Связность с Pack

- [ ] **L.1 cp/bh-упоминания актуальны:** все `cp.*` / `bh.*` из текста — в текущем PD.FORM.089
- [ ] **L.2 Понятия вводятся последовательно:** нет «вводится» без предшествующих prerequisites
- [ ] **L.3 Нет «воздуха»:** каждое предложение связано с предыдущим (нет случайных фактов)

## Уровень 🟢: Пилот-проверка (живой человек)

> Требует ≥3 пилотов, каждый оценивает по 6 критериям. Не автоматизируется.
>
> **Форма:** Issue `pilot-feedback` с шаблоном. Пилот отвечает на 6 вопросов — 6/6 = approve.

### 🟢 M. Пилот-тест (6 критериев)

Каждый пилот отвечает: **да / нет / частично**

- [ ] **M.1 Мем снят:** я понял, о чём речь, с первых 2-3 предложений
- [ ] **M.2 Метод применим:** я могу начать делать сегодня, инструкция понятна
- [ ] **M.3 Время соответствует:** практика заняла заявленное время (±20%)
- [ ] **M.4 Аналогия помогает:** я понял что-то новое через метафору
- [ ] **M.5 Нет барьеров:** текст читается без «что он имел в виду?»
- [ ] **M.6 Мировоззренческий сдвиг:** после подраздела я вижу ситуацию иначе

**Правило приёмки:** ≥3 пилота × 6/6 = `pilot-approved`. <3 пилотов или <6/6 = `needs-rework`.

## Алгоритм применения чек-листа

### Для агента (Claude Code):

```
1. Получить файл подраздела <subsection.md>
2. Уровень 🔴: запустить v4-lint (structure / porter / cross-guide / pack-drift / graph)
   - Если FAIL → остановиться, вернуть диагностику автору
   - Если PASS → перейти к 🟡
3. Уровень 🟡: прочитать текст, применить критерии G-L
   - Для каждого критерия: PASS / FAIL с цитатой из текста
   - Если FAIL → вернуть замечания автору
   - Если PASS → подраздел готов к пилот-тесту
4. Уровень 🟢: создать pilot-feedback issue, назначить пилотов
   - Результат приходит асинхронно
```

### Для CI (автоматика):

```yaml
# .github/workflows/v4-checklist-v1.yml
jobs:
  machine-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: 🔴 Structure
        run: python3 tools/v4-lint.py structure specs/v4-reference/
      - name: 🔴 Porter (staged files)
        run: python3 tools/v4-lint.py porter ${{ github.event.pull_request.changed_files }}
      - name: 🔴 Cross-guide
        run: python3 tools/v4-lint.py cross-guide specs/v4-reference/
      - name: 🔴 Pack-drift
        run: python3 tools/v4-lint.py pack-drift specs/v4-reference/ --pack ../PACK-personal/pack/.../PD.FORM.089-learner-rcs.md
      - name: 🔴 Graph
        run: python3 tools/v4-lint.py graph build specs/v4-reference/ --out-json /tmp/graph.json

  agent-check:
    needs: machine-check
    runs-on: ubuntu-latest
    steps:
      - name: 🟡 Агент-проверка (субагент Claude)
        # Вызывается вручную или через claude-code в CI
        run: echo "Требует Claude API — см. CHECKLIST-subsection-v1.md §🟡"
```
