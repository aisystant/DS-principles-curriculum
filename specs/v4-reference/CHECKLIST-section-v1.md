# Чек-лист проверки раздела руководства (v1)

> **Назначение:** агент Claude Code (или живой ревьюер) применяет этот чек-лист к разделу S после того, как все его подразделы SS прошли CHECKLIST-subsection-v1.
>
> **Применяется к:** одному разделу (S1, S2, ... SN), включающему 5-12 подразделов SS.
>
> **Когда запускать:**
> 1. После прохождения всех подразделов S через CHECKLIST-subsection-v1 (🔴+🟡)
> 2. При обнаружении проблем целостности раздела (например, при пилот-feedback «не понял переход между SS3 и SS4»)
> 3. После добавления/удаления подраздела в существующий раздел
>
> **Уровни:** 🔴 машина / 🟡 агент / 🟢 пилот. Правило вложенности: 🔴 100% PASS → 🟡 100% PASS → 🟢 ≥3 пилота × 6/6.
>
> **Иерархия чек-листов:** руководство → раздел (этот) → подраздел. Этот чек-лист НЕ дублирует проверки подраздела — он проверяет связность подразделов между собой и соответствие S общей дуге руководства.

---

## Уровень 🔴: Машинная проверка

> Запускается через `python3 tools/v4-lint.py section --id <section-id>`. Exit 0 = PASS.

### 🔴 A. Структурная полнота раздела

```bash
python3 tools/v4-lint.py section --id <section-id> specs/v4-reference/
```

- [ ] **A.1 Все SS присутствуют:** SS1...SSM согласно `structure-guide-N.md`, нет пропусков
- [ ] **A.2 Frontmatter раздела:** есть `section_id`, `title`, `parent_guide_id`, `stage_focus` (на каких ступенях фокус), `mastery_node` (узел мастерства, к которому раздел относится)
- [ ] **A.3 SS frontmatter согласован:** каждый SS имеет `parent_section_id`, совпадающий с этим S
- [ ] **A.4 README раздела:** короткое (1 экран) описание раздела с навигацией по SS

### 🔴 B. Связность prerequisites внутри раздела

```bash
python3 tools/v4-lint.py prerequisites-graph --scope section --id <section-id>
```

- [ ] **B.1 Prerequisites внутри раздела разрешаются:** если SS3.prerequisites = [SS1, SS2], то SS1 и SS2 действительно ДО SS3
- [ ] **B.2 Нет циклических зависимостей:** не бывает SS1 → SS2 → SS1
- [ ] **B.3 Cross-section prerequisites помечены:** если SS3 требует SS из другого S, ссылка вида `S2.SS5` (не просто `SS5`)

### 🔴 C. Pack-маппинг раздела

- [ ] **C.1 Все понятия введены в порядке:** в SS внутри раздела `introduces` идёт ДО `uses` (нет «использования до введения»)
- [ ] **C.2 Узел мастерства консистентен:** все SS раздела имеют `mastery_node`, совпадающий с разделом (нет «случайно затесавшегося» SS)
- [ ] **C.3 cp/bh-метрики раздела:** в `cp_check` / `bh_check` всех SS — измерения, релевантные узлу раздела

## Уровень 🟡: Агент-проверка (Claude Opus, context isolation)

> Субагент читает все SS раздела подряд (в порядке оглавления) и проверяет связность как целостного нарратива.

### 🟡 D. Нарративная связность подразделов

- [ ] **D.1 Логический переход между SS:** между соседними SS читатель понимает, почему дальше именно эта тема (не «случайный список»)
- [ ] **D.2 Никакой SS не «висит»:** каждый подраздел опирается на предыдущие в разделе или явно на другой S/руководство
- [ ] **D.3 Финальный SS закрывает раздел:** последний SS даёт явный итог раздела («что вы теперь умеете в этой зоне»)
- [ ] **D.4 Нет повторов:** одна и та же мысль/пример/аналогия не повторяется между SS раздела

### 🟡 E. Дуга по ступеням внутри раздела

- [ ] **E.1 stage_relevant подразделов согласован:** если раздел нацелен на ст. 2-3, то его SS не должны быть для ст. 5 (без явного flag `advanced: true`)
- [ ] **E.2 Тональность согласована:** все SS раздела используют тональность, соответствующую `stage_focus` раздела (бытовая для 1-2, системная для 3-5)
- [ ] **E.3 Сложность нарастает в правильном направлении:** SS внутри раздела идут от простого к сложному (или от частного к общему — но явно выбран один вектор)

### 🟡 F. Охват темы раздела

- [ ] **F.1 Все обещанные аспекты раскрыты:** заголовок и intro раздела обещают N аспектов — все N покрыты подразделами
- [ ] **F.2 Нет «дыры» в зоне раздела:** ключевые сценарии узла (по PD.FORM.089 рубрики) покрыты ≥1 SS
- [ ] **F.3 Нет «лишнего»:** ни один SS не выходит за заявленную тему раздела (если выходит — должен быть в другом S)

### 🟡 G. Аналогии и метафоры внутри раздела

- [ ] **G.1 Согласованность метафор:** если в разделе используется сквозная метафора (например, «болид = Пилот + Машина»), она применяется во всех релевантных SS, не противоречит сама себе
- [ ] **G.2 Нет конкурирующих метафор:** в разделе не должно быть двух метафор для одного и того же явления (читатель запутается)
- [ ] **G.3 Метафоры из одного семантического слоя:** не смешивать биологию + IT в одном разделе без явного перехода

### 🟡 H. Связь с другими разделами того же руководства

- [ ] **H.1 Ссылки на другие S корректны:** «см. S2.SS3» ведут на существующие подразделы
- [ ] **H.2 Нет противоречий:** утверждения в этом S не противоречат утверждениям в других S того же руководства
- [ ] **H.3 Раздел готовит к следующему:** если за этим S идёт логически связанный — финальный SS упоминает «следующий шаг — раздел N»

## Уровень 🟢: Пилот-проверка (живой человек на разделе целиком)

> Не для каждого подраздела отдельно, а после прохождения раздела полностью (типично 3-7 дней по 1 SS в день).

### 🟢 I. Пилот-тест раздела (6 критериев)

≥3 пилотов проходят раздел полностью, отвечают **да / нет / частично**:

- [ ] **I.1 Раздел как целое:** я могу одной фразой назвать, чему меня научил этот раздел
- [ ] **I.2 Логика перехода:** между подразделами не было «зачем сейчас это?»
- [ ] **I.3 Нет повторов:** не было ощущения «это уже было в предыдущем подразделе»
- [ ] **I.4 Завершённость:** к концу раздела я чувствую, что зона освоена (не «оборвалось на полуслове»)
- [ ] **I.5 Метафоры помогают:** сквозные образы раздела (если есть) поддерживают понимание, а не путают
- [ ] **I.6 Связь со ступенью:** материал раздела соответствует моей текущей ступени (или явно расширяет границу — это нормально)

**Правило приёмки:** ≥3 пилота × 6/6 → `section-approved`. <3 пилотов или <6/6 → `section-rework` с привязкой к конкретным SS.

## Алгоритм применения чек-листа

### Для агента (Claude Code):

```
1. Получить section_id (например, "guide-1-S5-roles")
2. ПРЕДУСЛОВИЕ: все SS раздела прошли CHECKLIST-subsection-v1 (🔴+🟡 PASS)
   Если нет — остановиться, попросить сначала закрыть SS-чек-листы
3. Уровень 🔴: запустить v4-lint section-mode (A-C)
   - FAIL → диагностика автору
4. Уровень 🟡: делегировать субагенту Claude Opus
   - Передать в prompt: все SS раздела + frontmatter раздела + этот чек-лист §🟡 (D-H)
   - Получить отчёт PASS/FAIL с цитатами
5. Уровень 🟢: запустить пилот-тест (≥3 пилотов, асинхронно)
   - Решение `section-approved` / `section-rework`
```

### Когда применять

| Триггер | Полный 🔴+🟡+🟢 | Только 🔴+🟡 | Только 🔴 |
|---------|-----------------|--------------|------------|
| Новый раздел в руководстве | ✅ | — | — |
| Добавление SS в существующий S | — | ✅ | — |
| Удаление SS из S | — | ✅ | — |
| Изменение порядка SS внутри S | — | ✅ | — |
| Hotfix typo в одном SS | — | — | ✅ (только этот SS) |

## Связь с другими чек-листами

- **CHECKLIST-subsection-v1.md** — предусловие (все SS раздела должны быть зелёные)
- **CHECKLIST-guide-v1.md** — следующий уровень (запускается после всех S руководства)

**Правило вложенности:**
```
1. Каждый SS → CHECKLIST-subsection-v1.md (🔴+🟡+🟢)
2. Затем S целиком → CHECKLIST-section-v1.md (🔴+🟡+🟢)  ← вы здесь
3. Затем руководство целиком → CHECKLIST-guide-v1.md (🔴+🟡+🟢)
```
