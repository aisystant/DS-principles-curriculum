# Индекс чек-листов руководства v4

> Три уровня детализации × три уровня контроля. Применяются в строгой иерархии: подраздел → раздел → руководство. На каждом уровне 🔴 → 🟡 → 🟢.

## Файлы

| Уровень | Файл | Применяется к | Размер |
|---------|------|---------------|--------|
| **Подраздел (SS)** | [CHECKLIST-subsection-v1.md](CHECKLIST-subsection-v1.md) | один SS (0.5-1.5K слов) | базовый, самый частый |
| **Раздел (S)** | [CHECKLIST-section-v1.md](CHECKLIST-section-v1.md) | один S (5-12 SS, 5-15K слов) | после всех SS |
| **Руководство** | [CHECKLIST-guide-v1.md](CHECKLIST-guide-v1.md) | целое руководство (5-10 S, 50-100K слов) | после всех S |

## Три уровня контроля (на каждом из трёх уровней детализации)

| Уровень | Что проверяет | Кто | Инструмент |
|---------|---------------|-----|------------|
| 🔴 **Машина** | Структура, frontmatter, ссылки, граф, drift | v4-lint | `python3 tools/v4-lint.py ...` |
| 🟡 **Агент** | Семантика, нарратив, тональность, связность | Claude Opus (context isolation) | субагент через Task tool |
| 🟢 **Пилот** | Восприятие, аналогия, мировоззренческий сдвиг | Живой человек | Issue-шаблон + 6 критериев |

## Правило вложенности (порядок применения)

```
ЗАПРЕЩЕНО запускать чек-лист уровня N+1, пока не закрыт уровень N.

1. Каждый SS → CHECKLIST-subsection-v1.md
   ├── 🔴 PASS обязателен
   ├── 🟡 PASS обязателен (для не-hotfix)
   └── 🟢 PASS желателен (≥3 пилота × 6/6)

2. Когда все SS раздела зелёные → CHECKLIST-section-v1.md
   ├── 🔴 PASS обязателен
   ├── 🟡 PASS обязателен
   └── 🟢 PASS желателен

3. Когда все S руководства зелёные → CHECKLIST-guide-v1.md
   ├── 🔴 PASS обязателен
   ├── 🟡 PASS обязателен
   └── 🟢 PASS желателен (для релиза руководства)
```

**Hotfix-исключение:** `[hotfix]` в коммите = только 🔴 на затронутом уровне (типично подраздел). 🟡 и 🟢 пропускаются — не для исправления typo.

## Когда какой чек-лист запускать

| Триггер | SS | S | Руководство |
|---------|-----|---|-------------|
| Создал новый SS | ✅ полный | — | — |
| Изменил content одного SS | ✅ полный | — | — |
| Добавил/удалил SS в S | ✅ затронутого SS | ✅ полный | — |
| Изменил структуру S (новый порядок SS) | — | ✅ полный | — |
| Создал новый S | ✅ всех SS | ✅ полный | — |
| Создал новое руководство | ✅ всех SS | ✅ всех S | ✅ полный |
| Изменил Pack (FORM/METHOD/CAT) | ✅ затронутых SS (🔴+🟡) | ✅ затронутых S (🔴+🟡) | ✅ только 🔴+🟡 |
| Hotfix typo | 🔴 только | — | — |
| Ежеквартальный аудит | 🔴+🟡 sample | 🔴+🟡 sample | ✅ полный |

## Алгоритм агента (Claude Code) при ревью

```
1. Определить уровень изменения (SS / S / руководство)
2. Открыть соответствующий CHECKLIST-{level}-v1.md
3. Запустить 🔴 проверки (CLI команды из чек-листа)
4. Если 🔴 PASS — делегировать 🟡 субагенту Claude Opus с context isolation
5. Если 🟡 PASS — создать issue для 🟢 пилот-теста (асинхронно)
6. Если на любом уровне FAIL — вернуть автору с диагностикой
7. После 🟢 PASS — merge в staging → main
```

## Когда чек-листы НЕ применяются

- **Hotfix typo** — только 🔴 на затронутом уровне
- **Эксперимент / черновик** в `drafts/` или `inbox/` — чек-листы не запускаются до перевода в `specs/v4-reference/`
- **Метаданные** (README руководства, version.json) — отдельная процедура, не чек-листы content

## Связь с PD.FORM.103 (конвейер разработки руководства)

Чек-листы покрывают этапы 6-9 конвейера ([PD.FORM.103](../../../../PACK-personal/pack/personal-development/02-domain-entities/formalizations/PD.FORM.103-guide-development-pipeline.md)):
- Этап 6 (🔴 lint) → CHECKLIST-*-v1.md §🔴
- Этап 7 (🟡 субагент) → CHECKLIST-*-v1.md §🟡
- Этап 8 (доставка в staging) → между 🟡 и 🟢
- Этап 9 (🟢 пилот) → CHECKLIST-*-v1.md §🟢

Этапы 1-5 (концепция → структура → Pack-маппинг → скелет → содержимое) — не покрываются чек-листами, они покрываются процессом WRITING-PIPELINE.
