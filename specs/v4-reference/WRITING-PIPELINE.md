# Конвейер написания подразделов универсальных руководств v4

## Общая схема

```
Граф понятий (Concept Map)
    ↓
Структура разделов → подразделов
    ↓
Написание подраздела (по промпту)
    ↓
Самопроверка (чек-лист агента)
    ↓
FPF-проверка (онтологическая чистота)
    ↓
Проверка субагентом (Claude)
    ↓
Commit → PR → Merge
```

---

## Этап 0. Граф понятий (Concept Map)

**Источник истины:** `0N-concept-map-guide-N.md`. Архитектурная рамка: `06-concept-graph-architecture.md`.

### Правила работы с понятиями

1. **Каждое понятие проходит через Concept Map перед попаданием в руководство.**
   - Если понятия нет в Concept Map — оно не попадает в руководство.
   - Исключение: понятие из Pack (FORM/METHOD) — добавляется в Concept Map, затем в руководство.

2. **Каждое понятие имеет стабильный ID** `PD.CON.NNN` (см. `06-concept-graph-architecture.md` §5.2). Текстовое имя может меняться, ID — нет.

3. **Каждое понятие связано с мета-мета-понятием.** Категории, критерии включения и тест-вопросы — см. Приложение А.

4. **Понятие не может висеть в воздухе** — у него должна быть:
   - Определение (1–3 предложения)
   - Связь с мета-мета-понятием
   - Связь с другими понятиями (родитель, дочернее, антоним)
   - Источник (Pack FORM/METHOD или обоснование добавления)

5. **Глубина упоминания понятия в подразделе размечается.** Понятие в строке «Понятия:» каждого подраздела сопровождается одним из четырёх маркеров:

   | Маркер | Что означает | Пример |
   |--------|--------------|--------|
   | `вводится` | Первое определение понятия в программе | `вводится: Болид (PD.CON.012)` |
   | `используется` | Применяется без переопределения | `используется: Машина — см. 1.S3.SS3` |
   | `противопоставлен` | Служит для различения | `противопоставлен: ассистент` |
   | `сослан` | Упомянут, отсылка на другой подраздел | `сослан: Защищённый слот → 1.S1.SS4` |

   Без маркера разметка считается некорректной. Контроль дублирования: один концепт = одно `вводится` во всей программе.

4. **Служебные/мета-термины не попадают в Граф понятий:**
   - Метрики: cp.*, bh.*
   - Операционные термины: Carry, drift, Week Close, MCP, FPF, DS, WP Gate
   - Технические термины: Git, хуки, алерты, Авария IWE
   - Аббревиатуры инфраструктуры: SoT, SoTA (как аббревиатуры)

### Алгоритм добавления нового понятия

```
1. Автор предлагает понятие → создаёт issue или заметку
2. Проверка: понятие теоретическое (не служебное)?
3. Проверка: нет ли дубля в Графе (одно слово = одно понятие)?
4. Связь с мета-мета-понятием — указать категорию
5. Связь с существующими понятиями — родитель, дочернее, антоним
6. Утверждение пилотом или методологом
7. Запись в Concept Map
8. Только после этого — использование в руководстве
```

## Этап 1. Структура: от Графа к подразделам

### Распределение понятий

- **Раздел** = тематическая неделя (~7 подразделов)
- **Подраздел** = дневная доза (1–3 понятия + практика)
- **Плотность:** 1–3 понятия на подраздел. Не больше.

### Правила распределения

1. **Понятия группируются по связям**, а не произвольно.
   - Плохо: «Метод + Болид + Эмерджентность» в одном подразделе
   - Хорошо: «Культура ученика + Допуск к сложности» (родитель-дочернее)

2. **Первое появление понятия = его определение.**
   - Повторные появления — только краткая напоминание + ссылка.

3. **Кейсы и истории — не понятия.**
   - «Эффект Земмельвейса» — кейс, не понятие. В тексте как пример, в сигнатуре не указывается.

## Этап 2. Промпт написания подраздела

**Роль:** Мыслитель-практик, который думает вслух. Не отстранённый эксперт и не коуч.

**Тон:** Инструктивный: «Вот как устроено» — не «вот что ты должен». Личный опыт вместо советов, вопрос вместо императива.

### Промпт (копируется для каждого подраздела)

```
Напиши подраздел руководства по личному развитию.

Параметры:
- Руководство: [1/2/3/4]
- Раздел: [номер + название]
- Подраздел: [номер + название]
- Время: [X мин чтение + Y мин практика]
- Целевой объём: 500–1500 слов

Понятия этого подраздела (из Графа понятий):
- Задаётся: [понятие 1] — [краткое определение]
- Задаётся: [понятие 2] — [краткое определение]
- Используется: [понятие 3] — [где впервые определено]

Ограничения (строго):
1. НЕ использовать шифры Pack в тексте: не писать PD.METHOD.021, FORM.006, cp.wld, bh.awr
2. НЕ использовать служебные термины: drift, gate, bottleneck как жаргон
3. НЕ использовать заголовки "Развёртка", "Суть", "Теория" — текст идёт монолитными абзацами
4. НЕ писать про IWE в руководствах 1 и 2 (участники ещё не знакомы)
5. НЕ использовать образовательную лексику: школа, студенты, преподаватель, обучение
6. "В одном предложении" в начале должно быть понятно широкой аудитории — без спецтерминов
7. Сигнатура понятий: только теоретические понятия из Графа, не кейсы
8. Степени мастерства — без ссылки на FORM.006, просто 4 степени
9. Проверка себя — на человеческом языке, без шифров
10. Каждое утверждение должно иметь источник (цитата из Pack или пример из жизни)

Структура (обязательная последовательность):
1. "В одном предложении" — суть, понятная без контекста
2. Сигнатура понятий (Задаются / Используются)
3. Определение из источника — цитата 1–2 абзаца
4. Развитие мысли — 2–4 абзаца: как устроено, почему работает
5. Пример из жизни — конкретный, с деталями (имя, профессия, ситуация)
6. Типичная ошибка / антипаттерн — что делают не так
7. Степени мастерства — таблица 4 степеней
8. Проверка себя — 2–3 вопроса на человеческом языке
9. На практике — что делать (без упоминания IWE для рук. 1–2)
10. См. также — кросс-ссылки на другие разделы

Стилистика:
- Предложения длинные, 15–40 слов
- Абзацы крупные: 4–8 предложений
- Одна развёрнутая аналогия из бытовой жизни
- Один пример из личного опыта или типичного опыта участников
- Один противопоставление для различения
- Жирный текст для ключевых тезисов внутри абзаца
```

## Этап 3. Чек-лист самопроверки (агент выполняет перед коммитом)

### Структурная проверка

- [ ] Подраздел открывается "В одном предложении", понятном без контекста
- [ ] Сигнатура понятий содержит только теоретические понятия из Графа
- [ ] Кейсы/истории НЕ в сигнатуре понятий
- [ ] Все понятия из сигнатуры есть в Графе понятий (или в списке на утверждение)
- [ ] Каждое утверждение имеет источник (цитата или пример)
- [ ] Пример конкретный: имя/профессия/ситуация, не абстрактный
- [ ] Типичная ошибка описана как антипаттерн, а не как запрет

### Языковая проверка

- [ ] Нет шифров Pack в тексте: PD.METHOD.021, FORM.006, cp.wld, bh.awr и т.д.
- [ ] Нет служебных терминов: drift, gate, bottleneck как жаргон
- [ ] Нет заголовков "Развёртка", "Суть", "Теория"
- [ ] Нет образовательной лексики: школа, студенты, преподаватель, обучение
- [ ] Название подраздела понятно широкой аудитории
- [ ] "В одном предложении" понятно без спецтерминов
- [ ] Степени мастерства без ссылки на FORM.006
- [ ] Проверка себя на человеческом языке, без шифров
- [ ] Для руководств 1 и 2 — нет упоминания IWE в тексте

### Стилистическая проверка

- [ ] Тон: "Вот как устроено", не "вот что ты должен"
- [ ] Предложения длинные, 15–40 слов
- [ ] Абзацы крупные, 4–8 предложений
- [ ] Есть развёрнутая аналогия из бытовой жизни
- [ ] Есть пример из жизни созидателя
- [ ] Есть противопоставление для различения
- [ ] Жирный текст для ключевых тезисов внутри абзаца
- [ ] Нет воды, нет "важно отметить", нет "стоит рассмотреть"

### Метрики

- [ ] Объём: 500–1500 слов
- [ ] Время чтения + практика соответствует норме руководства
- [ ] Кросс-ссылки в конце, не в середине текста

## Этап 4. FPF-проверка

**Режим:** review (сверка артефакта с принципами)

### Проверяемые паттерны

| Паттерн | Что проверяем | Критерий |
|---------|---------------|----------|
| **A.7 Strict Distinction** | Нет смешения Object/Description/Role/Work | Понятие ≠ кейс, определение ≠ пример, автор ≠ персонаж |
| **A.11 Ontological Parsimony** | Минимум понятий, нет дублей | 1–3 понятия на подраздел, одно слово = одно понятие |
| **A.10 Evidence Graph** | Утверждения привязаны к источнику | Каждое утверждение — или цитата Pack, или пример из жизни, или исторический факт |
| **A.6 Boundary Discipline** | Чёткие границы между понятиями | "Культура ≠ мотивация" — различение явное, не размытое |
| **Thinking Through Writing** | Мышление зафиксировано в форме | Текст не "вода", каждый абзац несёт мысль |
| **A.1.1 Bounded Context** | Термины не убегают за границы руководства | Понятие IWE не используется в рук. 1–2, понятие "Болид" не разъясняется в рук. 2 |

### Алгоритм FPF-проверки

```
1. Прочитать подраздел
2. Для каждого понятия: проверить A.7 (что это — object, description, role?)
3. Проверить A.11: сколько понятий? есть ли дубли?
4. Проверить A.10: каждое утверждение — есть ли источник?
5. Проверить A.6: различения чёткие? границы ясные?
6. Проверить Bounded Context: термины в своих руководствах?
7. Результат: PASS / WARN (исправить) / FAIL (переписать)
```

## Этап 5. Проверка субагентом (Claude)

**Роль субагента:** Внешний рецензент с context isolation.

### Что передаётся субагенту

1. **Ссылка** на файл подраздела (GitHub permalink)
2. **Контекст:**
   - Какое руководство, какой раздел
   - Какие понятия должны быть в подразделе (из Графа)
   - Какие требования применяются (из 00-requirements.md)
3. **Вопросы для проверки:**
   - Понятен ли текст широкой аудитории?
   - Есть ли в тексте шифры или служебные термины?
   - Корректны ли примеры?
   - Не нарушены ли границы руководства?

### Формат ответа субагента

```
- Статус: ✅ PASS / ⚠️ WARN / ❌ FAIL
- Найденные проблемы (с указанием строк)
- Рекомендации по исправлению
```

### Когда вызывать субагента

- Обязательно: для первого подраздела каждого руководства (пилот)
- Обязательно: если FPF-проверка дала WARN/FAIL
- Рекомендуется: каждый 5-й подраздел (выборочный контроль)

## Приложение А. Мета-мета-понятия (понятия трансдисциплин)

**Источник истины:** [SPF.SPEC.002 § 2 «Базовая онтология SPF»](../../../DS-Knowledge-Index-Tseren/docs/SPF/ontology.md). Мета-мета-понятия — это понятия трансдисциплин (FPF Part A), переведённые на русский в SPF. Любое доменное понятие ЛР обязано иметь родителя из этого списка (правило SPF.SPEC.002 § 5 п.2). Владелец расширения — Knowledge Extractor (DP.AISYS.013).

**Доменный Граф понятий ЛР:** [PACK-personal/ontology.md § 2 «Глоссарий домена»](../../../PACK-personal/ontology.md) — там каждое понятие ЛР уже привязано к родителю U.*.

**Релевантные для ЛР мета-мета-понятия (выписка из SPF.SPEC.002 § 2):**

| FPF/SPF ID | Понятие (рус.) | Что фиксирует | Типичные домены ЛР |
|------------|----------------|---------------|--------------------|
| `U.System` | Система | Холон с границей, взаимодействующий с окружением | Созидатель, Творческий конвейер, IWE, Болид (как аналогия) |
| `U.Episteme` | Эпистема | Единица знания/описания как артефакт | Мировоззрение, Active Inference, Эпистемический граф, Формализация |
| `U.Method` | Метод | Способность производить рабочий продукт | Учёт времени, Защищённый слот, Каскад ВДВ, Гигиенический минимум, Knowledge Extraction |
| `U.MethodDescription` | Описание метода | Рецепт/инструкция метода | Протоколы ОРЗ, Шаблоны письма |
| `U.RoleAssignment` | Назначение роли | Контекстная привязка роли | Ученик, Интеллектуал, Профессионал, Исследователь, Просветитель |
| `U.Capability` | Способность | Диспозиционное свойство | Знание, Навык, Мастерство |
| `U.Characteristic` | Характеристика | Измеряемая ось оценки | Калибр личности, Агентность, Собранность, cp.*/bh.* |
| `U.Flow` | Поток | Эволюция ограничений | Состояние, Три потока личной жизни |
| `U.BoundedContext` | Ограниченный контекст | Семантическая рамка со словарём | Pack, Bounded Context руководства, Домен деятельности |
| `U.Boundary` | Граница | Разделение внутреннего и внешнего | Граница системы, Граница ответственности |
| `U.Kind` | Вид/Тип | Классификационная единица | Ступень развития, Тип роли, Квалификация |
| `U.Work` | Работа | Запись свершившегося действия | Рабочий продукт, Записанная сессия, Day Close |
| `U.WorkPlan` | План работ | Расписание намерений | WeekPlan, DayPlan, Стратегия |
| `U.Commitment` | Обязательство | Деонтический объект | Намерение, Ценность как принцип |
| `U.ServiceClause` | Обещание сервиса | Контракт обслуживания | Обещание IWE пилоту, Обещание метода |

**Применение в концепт-мапе:** для каждого доменного понятия указывать в строке `parent: U.*`. Если родителя из этого списка нет — кандидат на расширение SPF (Knowledge Extractor).

**Правило «понятие vs кейс vs аналогия»** — см. `06-concept-graph-architecture.md` § 8. Кейсы (Эффект Земмельвейса, Хохланд) и аналогии (Болид, асептика для хирурга) НЕ получают `parent: U.*` и НЕ попадают в строку «Понятия:» подраздела — они в тексте как иллюстрация/метафора.

## Приложение Б. Список служебных терминов (не использовать в тексте руководства)

| Термин | Почему служебный | Чем заменить в тексте |
|--------|------------------|----------------------|
| cp.* / bh.* | Метрики, не понятия | «осознанность», «стабильность», «инвестирование" |
| drift / drift-detection | Операционный термин | «расхождение", «отставание" |
| gate | Технический термин | «ворота", «переход", «порог" |
| bottleneck | Технический термин | «узкое место", «застревание" |
| Carry / carry-over | Операционный термин | «перенос", «остаток" |
| Авария IWE | Операционный термин | «сбой", «потеря фокуса" |
| MCP, FPF, DS | Аббревиатуры инфраструктуры | Не использовать в руководствах 1–2 |
| WP Gate | Операционный термин | «проверка перед началом" |
| Week Close / Month Close | Процедурные термины | «закрытие недели", «подведение итогов" |
| Git / хуки | Технические термины | Не использовать в руководствах 1–2 |
| PD.METHOD.021, FORM.006 | Шифры Pack | Не использовать в тексте; фиксировать в служебном frontmatter |
