## Общая схема

```
Контекст (WP-245, Паки, текущие руководства, role-prefixes)
    ↓
Технический gate (валидация структуры v4-reference)
    ↓
Структура разделов → подразделов (mastery_node, stage_relevant)
    ↓
Написание подраздела (промпт: мем → метод → мировоззрение)
    ↓
Самопроверка (чек-лист агента)
    ↓
Платформенная сверка (bh/cp vs FORM.089 v4)
    ↓
FPF-проверка + нарративный gate
    ↓
Porter-readiness gate
    ↓
Проверка субагентом
    ↓
Кросс-руководственная согласованность
    ↓
Пилот-тест с живым пользователем (WP-245 Ф28.3)
    ↓
Верификация времени (секундомер)
    ↓
Commit → PR → Merge
    ↓
Протокол сопровождения (maintenance)
```

## Этап 1. Контекст и входные данные

**Принцип:** граф понятий — производный артефакт, генерируемый из метаданных подраздела (`06-concept-graph-architecture.md` §4). Автор работает с контекстом программы ЛР, Паками, текущими руководствами и ролевой траекторией. Граф обновляется параллельно с письмом или после — но **не блокирует** его. Понятие, обнаруженное в процессе письма, фиксируется в графе ретроспективно.

### 1.1. Что уже построено (текущие руководства и программы)

Перед написанием нового подраздела автор обязан знать, что уже существует:

- **Текущие руководства** (`aisystant/docs/docs/ru/personal-design/`) — устаревшие версии программы ЛР, которые v4 заменяет. Проверить: не дублирует ли новый подраздел то, что уже есть в старых руководствах.
- **РП245 (Программа личного развития)** — единый нарратив: дуга «от собранности к созиданию», два столпа (мировоззрение + методы), механика «мем → метод → мировоззрение».
- **РП300 (Универсальные руководства)** — реестр понятий (`07-concept-candidates-v4.md`), структура 4 руководств, concept maps, mapping старого→нового (`05-mapping-old-to-new.md`).
- **Рабочее развитие** — программа, следующая за ЛР. Каждое руководство v4 должно иметь в конце раздел «Что дальше», указывающий, какие методы и навыки переходят туда.

### 1.2. Паки как источник (PACK-personal)

**Источник истины онтологии:** `PACK-personal/ontology.md` §2 — глоссарий домена ЛР.

Автор перед написанием подраздела проверяет:
- Есть ли понятия подраздела в `PACK-personal/ontology.md`?
- Если нет — в `07-concept-candidates-v4.md`?
- Какой у них parent `U.*` из SPF.SPEC.002 §2?
- Какой Pack-источник (`PD.FORM.NNN` / `PD.METHOD.NNN`)?

**Правило:** каждое утверждение в тексте должно иметь источник в Pack. Не выдумывать определения — брать из FORM/METHOD/CAT.

### 1.3. Онтология Созидателя (PD.FORM.089 v4)

- **Три узла развития** — мыслительное мастерство, мастерство саморазвития, мастерство управления IWE.
- **Пять ступеней** — Случайный → Практикующий → Систематический → Дисциплинированный → Проактивный.
- **bh/cp-пороги** — К4 fix: `bh.inv` считается только по `slot_logged.hours` + `lesson_completed.duration_minutes/60`.
- **Ролевая траектория** — Ученик → Интеллектуал → Профессионал → Исследователь → Просветитель.

### 1.4. Правила работы с понятиями

1. **Каждое понятие идентифицируется тройкой:** каноническое русское имя + родитель `U.*` из SPF.SPEC.002 §2 + источник Pack (`PD.FORM.NNN` / `PD.METHOD.NNN`). Локальный `PD.CON.NNN` **не вводится**.
2. **Глубина упоминания размечается:**

   | Маркер | Что означает | Пример |
   |--------|--------------|--------|
   | `вводится` | Первое определение в программе | `вводится: Созидатель → U.System` |
   | `используется` | Применяется без переопределения | `используется: Машина — см. 1.S3.SS3` |
   | `противопоставлен` | Служит для различения | `противопоставлен: ассистент` |
   | `сослан` | Упомянут, отсылка на другой подраздел | `сослан: Защищённый слот → 1.S1.SS4` |

3. **Служебные термины не попадают в Граф:** `cp.*`, `bh.*`, drift, gate, bottleneck, MCP, FPF, DS, Git, хуки, Авария IWE и т.д. См. Приложение Б.

### 1.5. Правило «Pack-first» (блокирующее)

> **Если понятия нет в Pack — его нет в руководстве.**

| Сценарий | Что делать |
|---|---|
| Перед началом письма SS | Автор проверяет: все понятия из `introduces` есть в `PACK-personal/ontology.md` §2 + имеют `PD.FORM.NNN` / `PD.METHOD.NNN` / `PD.WP.NNN`. Если нет — **спринт по Pack, не письмо.** |
| В процессе письма появилось новое понятие | **Стоп.** Не вносить в текст руководства. Зафиксировать в `07-concept-candidates-v4.md` → передать Knowledge Extractor (DP.AISYS.013) → дождаться появления в `PACK-personal/ontology.md` §2 + назначения `U.*` и Pack-источника → только потом добавить в `introduces` и текст. |
| Понятие есть в старом руководстве, но нет в Pack | Старое руководство — не источник истины. Извлечь определение, формализовать в Pack, потом использовать в v4. |
| Понятие есть в Concept Graph / candidates, но не в ontology.md §2 | Concept Graph и candidates — **производные артефакты**, не источники истины. Ждать формализации Knowledge Extractor'ом. |

**Исключение:** бытовые аналогии и кейсы (не теоретические понятия) можно использовать без Pack-источника, но они **не попадают** в `introduces`/`uses` и не становятся концепт-нодами графа.

**Почему это блокер:** `v4-lint.py` Этап 7 (A.10 Evidence Graph) даст FAIL, если `introduces` содержит понятие без Pack-источника. Портной не сможет построить маршрут по orphan-концепту. При обновлении Pack произойдёт неотслеживаемый drift.

## Этап 2. Технический gate

Перед началом массового написания — валидация технической целостности `v4-reference/`.

### Что проверяется

- [ ] **Парсер Guide 4** — формат «Понятия:» не обрезается (нет `(` в начале имени). См. `08-v4-corrections-roadmap.md` 🔴.
- [ ] **Численная сортировка** — разделы идут S1, S2, …, S10 (не S1, S10, S2). См. `CONCEPT-GAP-ANALYSIS.md`.
- [ ] **Нет пропусков SS** — каждый раздел имеет SS1–SS7 или явно помечен `optional: true`.
- [ ] **Омонимы разрешены** — нет одного слова с разным parent `U.*` без квалификатора. См. `06-concept-graph-architecture.md` §9.

**Инструмент:**

```bash
python3 tools/v4-lint.py structure specs/v4-reference/
```

Запускается локально автором и автоматически на PR через [.github/workflows/v4-lint.yml](../../.github/workflows/v4-lint.yml). Exit 0 = PASS, exit 1 = FAIL с диагностикой. Реализация: [tools/v4-lint.py](../../tools/v4-lint.py). Контракт инструмента: [WP-321](../../../DS-my-strategy/inbox/WP-321-v4-lint-validators.md).

## Этап 3. Структура: от контекста к подразделам

### Контракт подраздела (frontmatter)

```yaml
---
subsection_id: PD.GUIDE.1.S1.SS1
title: "Культура ученика"
order: 1
mastery_node: [саморазвитие]          # узел из Приложения В
stage_relevant: [1, 2]                # ступени, где доступен
introduces: ["Стиль жизни ученика", "Культура ученика"]
uses: ["Созидатель — см. PD.GUIDE.1.S3.SS1"]
prerequisites: []
can_do:
  - "Могу объяснить разницу между «проектом» и «стилем» применительно к своей практике"
cp_check: [cp.wld]
bh_check: [bh.awr]
bottleneck_hint: "Если застрял — проверь cp.wld: воспринимаешь ли развитие как норму, а не как проект"
---
```

**Правила заполнения:**
- `mastery_node` — один или несколько из: `мыслительное`, `саморазвитие`, `iwe`. См. Приложение В.
- `stage_relevant` — ступени 1–5. Определяет тональность и фильтрацию Портным.
- `introduces` — только каноническое имя, без `U.*` (это фаза 2).
- `can_do` — начинается с «Могу», наблюдаемое действие, проверяемое за ≤5 мин.
- Кейсы и аналогии **не попадают** в `introduces`/`uses`.

### Файловая структура

Каждое руководство организовано в **подпапки по разделам**, как в Guide 1:

```
docs/ru/personal-design/1-2-self-development-methods/
├── s1-method-form-culture/
│   ├── 1.01.md
│   ├── 1.02.md
│   └── ...
├── s2-time-investment/
│   ├── 2.01.md
│   └── ...
├── s6-leisure-organization/
│   ├── 6.01.md
│   └── ...
```

**Правило (блокирующее):** подразделы Guide 2–4 размещаются **только в подпапках разделов**. Плоское размещение файлов в корне директории руководства (`1-2-self-development-methods/6.01.md`) запрещено — это ломает навигацию, URL-конвенцию и сборку оглавления.

| Что | Где |
|-----|-----|
| Guide 1 | `1-1-systemic-self-development/sN-section-name/` |
| Guide 2 | `1-2-self-development-methods/sN-section-name/` |
| Guide 3 | `1-3-iwe-work-and-development/sN-section-name/` |
| Guide 4 | `1-4-systems-thinking-introduction/sN-section-name/` |

### Распределение понятий

- **Раздел** = тематическая неделя (~7 подразделов)
- **Подраздел** = дневная доза (1–3 понятия + практика)
- **Плотность:** 1–3 понятия на подраздел. Не больше.

### Правила

1. **mastery_node** — каждый подраздел обязан относиться к хотя бы одному узлу (Приложение В).
2. **stage_relevant** — явная метка, на каких ступенях подраздел доступен.
3. **Понятия группируются по связям**, а не произвольно.
4. **Первое появление понятия = его определение.** Повторные — только краткая напоминание + ссылка.
5. **Кейсы и истории — не понятия.** См. `06-concept-graph-architecture.md` §8.

## Этап 4. Промпт написания подраздела

**Роль:** Мыслитель-практик, который думает вслух.

**Тон:** Инструктивный: «Вот как устроено» — не «вот что ты должен». Вопрос вместо императива. Наблюдение и обобщение вместо личных историй автора (запрещены, см. 00-requirements.md § «Запрет личного опыта автора»).

### Промпт

```
Напиши подраздел руководства по личному развитию.

Параметры:
- Руководство: [1/2/3/4]
- Раздел: [номер + название]
- Подраздел: [номер + название]
- Время: [X мин чтение + Y мин практика]
- Целевой объём: 500–1500 слов
- mastery_node: [мыслительное / саморазвитие / iwe]
- stage_relevant: [N–M]

Понятия этого подраздела:
- Задаётся: [понятие 1] — [краткое определение]
- Задаётся: [понятие 2] — [краткое определение]
- Используется: [понятие 3] — [где впервые определено]

Нарративная дуга (обязательно):
- Мем (блокирующее убеждение): «...»
- Метод (что вводится): «...»
- Мировоззрение (как меняется картина): «...»
- Связь с горизонтом цивилизации: [только для ступеней 3–5]

Ограничения (строго):
1. НЕ использовать шифры Pack: PD.METHOD.021, FORM.006, cp.wld, bh.awr
2. НЕ использовать служебные термины: drift, gate, bottleneck как жаргон
3. НЕ использовать заголовки "Развёртка", "Суть", "Теория"
4. НЕ писать про IWE в руководстве 1 (ввод мировоззрения без инструментов). В руководстве 2 — IWE как контекст применения метода (усиление, не замена).
5. НЕ использовать образовательную лексику: школа, студенты, преподаватель, обучение
6. "В одном предложении" — понятно широкой аудитории, без спецтерминов
7. Сигнатура понятий: только теоретические, не кейсы
8. Степени мастерства — без FORM.006, просто 4 степени
9. Проверка себя — на человеческом языке, без шифров
10. Каждое утверждение — источник (цитата Pack или пример из жизни)
11. Тональность соответствует stage_relevant: ст. 1–2 — конкретный первый шаг, без
    цивилизационных обобщений; ст. 3–5 — можно связь с горизонтом
12. Мем → Метод → Мировоззрение: цепочка должна быть видна

Структура (обязательная последовательность), 14 блоков:
1. **Время** — чтение + практика = сумма (обязательно перед открытием)
2. "В одном предложении" — мем + объект + сдвиг
3. Сигнатура понятий
4. Мем — блокирующее убеждение (1–2 абзаца)
5. Определение из источника — цитата 1–2 абзаца
6. Развитие мысли — 2–4 абзаца
7. Метод — что делать (минимальный шаг)
8. Пример из жизни — конкретный, с деталями
9. Типичная ошибка / антипаттерн
10. Степени мастерства — 4 степени (нумерованный список)
11. Проверка себя — 2–3 вопроса
12. На практике — что делать
13. См. также — кросс-ссылки на другие разделы
14. Что дальше — связь с Рабочим развитием (для руководств 1–4)

> **Эволюция v4.1 (2026-05-18):** блок "Что узнаешь" удалён как дубль "В одном предложении". Блок "Время" вынесен в отдельный элемент структуры. Степени мастерства — нумерованный список из 4 пунктов, не таблица.

Стилистика:
- Предложения длинные, 15–40 слов
- Абзацы крупные: 4–8 предложений
- Одна развёрнутая аналогия
- Один обобщённый кейс (без имени автора, без личной истории — обезличенный пример или типовая ситуация)
- Один противопоставление
- Жирный текст для ключевых тезисов

Запрещено:
- Истории от первого лица автора («я как-то...», «помню, в 2019 году...», «однажды я заметил...»)
- Любая биографическая привязка к фигуре автора
- Прямые цитаты «реальных людей» с именами
```

## Этап 5. Чек-лист самопроверки (агент выполняет перед коммитом)

### Структурная проверка

- [ ] Подраздел открывается "В одном предложении", понятном без контекста
- [ ] Сигнатура содержит только теоретические понятия (нет кейсов/аналогий)
- [ ] Все понятия есть в 07-candidates или PACK-personal/ontology.md
- [ ] Есть явный **мем** (блокирующее убеждение), который снимается
- [ ] Есть **метод**, который вводится
- [ ] Есть связь с **мировоззрением** (как меняется картина)
- [ ] Цепочка «мем → метод → мировоззрение» прослеживается
- [ ] Каждое утверждение имеет источник
- [ ] Пример конкретный: имя/профессия/ситуация

### Нарративная проверка

- [ ] Тональность соответствует `stage_relevant`: ст. 1–2 — конкретика, без цивилизационных обобщений; ст. 3–5 — можно горизонт
- [ ] `mastery_node` указан корректно
- [ ] Нет омонимов без квалификатора
- [ ] Кейсы и аналогии — только в тексте, не в сигнатуре

### Языковая проверка

- [ ] Нет шифров Pack в тексте
- [ ] Нет служебных терминов
- [ ] Нет образовательной лексики
- [ ] Название понятно широкой аудитории
- [ ] Степени мастерства без FORM.006
- [ ] Проверка себя на человеческом языке
- [ ] Для рук. 1 — нет IWE. Для рук. 2 — IWE как контекст применения метода.

### Стилистическая проверка

- [ ] Тон: "Вот как устроено"
- [ ] Предложения 15–40 слов
- [ ] Абзацы 4–8 предложений
- [ ] Есть аналогия, пример, противопоставление
- [ ] Нет воды, нет "важно отметить"

### Метрики

- [ ] Объём 500–1500 слов
- [ ] Время чтения + практика = норме руководства
- [ ] Кросс-ссылки в конце, не в середине

## Этап 6. Платформенная сверка (pre-commit gate)

**Зачем:** текст не должен ссылаться на устаревшие cp/bh-пороги или неправильное определение `bh.inv`.
**Когда:** каждый коммит изменения подраздела. Связан с Этапом 14 (Maintenance) — там же используется тот же валидатор, но в режиме периодической ревизии всего корпуса при изменении Pack.

### Инструмент

```bash
python3 tools/v4-lint.py pack-drift specs/v4-reference/ \
  --pack ../PACK-personal/pack/personal-development/02-domain-entities/formalizations/PD.FORM.089-learner-rcs.md
```

## Этап 7. FPF-проверка + нарративный gate

**Режим:** review (сверка артефакта с принципами).

### Разделение автоматизируемого и семантического

Часть FPF-паттернов проверяется механически валидатором (см. ниже), часть требует семантического чтения субагентом (Этап 9). Это разделение фиксирует границу инструмента: линтер ловит структурные нарушения, субагент — нарушения смысла.

| Паттерн | Где проверяется | Критерий |
|---------|-----------------|----------|
| **A.7 Strict Distinction** | v4-lint structure (`check_cases_in_introduces`) | Понятие ≠ кейс — кейсы (Земмельвейс, Хохланд) не попадают в `вводится` |

| **A.10 Evidence Graph** | v4-lint structure (`check_evidence_graph`) | Каждое `вводится` имеет источник Pack `(PD.FORM.NNN)` |
| **Тройка идентификации** | v4-lint structure (`check_triple_identification`) | Имя + `parent U.*` + Pack source — всё на месте |
| **A.1.1 Bounded Context** | v4-lint porter (`mastery_node:iwe` в guide 1) | IWE не в рук. 1; в рук. 2 — IWE как контекст применения метода |
| **A.6 Boundary Discipline** | Этап 9 (субагент) | Различение явное, граница чёткая |
| **Thinking Through Writing** | Этап 9 (субагент) | Каждый абзац несёт мысль |
| **Нарратив «мем→метод→мировоззрение»** | Этап 9 (субагент) | Цепочка прослеживается |
| **Дуга по ступени** | Этап 9 (субагент) | Тональность соответствует `stage_relevant` |

### Инструмент (автоматизируемая часть)

```bash

### Алгоритм (на стыке автомата и человека)

```

```

**Роль субагента:** Внешний рецензент с context isolation. Покрывает паттерны, которые линтер из Этапа 7 не может проверить (требуют семантического чтения, не структурного).

**Зачем:** 40 мин — инженерный параметр, не декларация.

## Этап 13. Фиксация и доставка

### Commit

Куда коммитить — зависит от слоя:

| Слой | Что | Куда | Команда |
|------|-----|------|---------|
| Структура (frontmatter + строка «Понятия:») | `NN-structure-guide-N.md` | `DS-principles-curriculum/specs/v4-reference/` | `git add specs/v4-reference/NN-structure-guide-N.md` |

git -C ~/IWE/DS-IT-systems/aisystant commit -m "feat(personal-design): subsection X.Y — [название]"
```

## Этап 14. Протокол сопровождения (Maintenance + Pack-watcher trigger)

После публикации текст устаревает. Нужен механизм периодической ревизии. **Отличие от Этапа 6** (pre-commit gate): Этап 6 ловит дрейф на каждом коммите, Этап 14 — реактивная ревизия всего корпуса при изменении Pack.

### Триггеры обновления

| Событие | Что проверяется | Действие |
|---------|-----------------|----------|
| **Новый FORM/METHOD** | Подразделы, ссылающиеся на изменённый артефакт | Пометить «требует проверки» |
| **Изменение cp/bh-рубрик** (FORM.089) | Все подразделы с `cp_check`/`bh_check` | Запустить `v4-lint pack-drift`, пересчитать `stage_relevant` |
| **Новое различение в Pack** | Подразделы со смежными понятиями | Проверить A.6 Boundary Discipline |
| **Дрейф concept-graph** | `CONCEPT-GAP-ANALYSIS.md` | Запустить `v4-lint structure` + `v4-lint graph diff` |
| **Pack-watcher** (CD-PIPELINE.md Спринт 1) | Коммит в `PACK-personal/.../PD.FORM.089` | Авто-issue в `DS-principles-curriculum` |

### Процедура

1. **Детекция:** валидатор сравнивает упоминания `cp.*` / `bh.*` в текстах и frontmatter с актуальной FORM.089:

   ```bash
   python3 tools/v4-lint.py pack-drift specs/v4-reference/ \
     --pack ../PACK-personal/pack/personal-development/02-domain-entities/formalizations/PD.FORM.089-learner-rcs.md
   ```

   Запускается при Week Close (Security Audit Cadence паттерн) и после каждого обновления FORM.089. Реализация: [tools/v4-lint.py](../../tools/v4-lint.py).
2. **Пометка:** подразделы, требующие проверки, получают label `needs-review` в frontmatter.
3. **Перепроверка:** автор проходит этапы 5–9 для помеченных подразделов.
4. **Версионирование:** minor bump (v4.1, v4.2) при изменении понятий; patch (v4.0.1) при стилистических правках.
