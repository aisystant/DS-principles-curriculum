# DS-principles-curriculum — Инструкции для Claude Code

> **Тип репо:** Downstream/surface
> **Upstream Pack:** PACK-education (EDU) — методика обучения
> **Upstream Content:** ZP (нулевые принципы), FPF (первые принципы)
> **Объект:** Конкретные программы обучения принципам мышления

---

## 1. Что это

Конкретное **применение** методов из PACK-education (EDU) к обучению **нулевым принципам (ZP)** и **первым принципам (FPF)**. Pack EDU описывает КАК учить (абстрактно). Этот репо описывает КАК учить ПРИНЦИПАМ (конкретно).

**Ключевое различие:**

| Pack EDU | DS-principles-curriculum |
|----------|--------------------------|
| Шаблон ячейки (8 пустых полей) | Заполненные ячейки (ZP.1@1, ZP.1@2, ...) |
| TPM — абстрактная модель профиля | Конкретный профиль ученика по ZP/FPF |
| Диагностический тест — методология | Банк вопросов по ZP/FPF |
| Конструктор программ — алгоритм | Готовые программы для аудиторий |

### Двухуровневые ячейки (FPF)

FPF содержит ~216 паттернов в 7 частях (A-G). Для curriculum используется двухуровневая структура:

| Уровень | Файл | Что описывает |
|---------|------|--------------|
| **Макро-ячейка** | `FPF-A-kernel.md` | Часть FPF целиком (5 глубин Bloom) |
| **Микро-ячейка** | `FPF-A/A.1-holon.md` | Конкретный паттерн внутри Part (5 глубин) |

Макро-ячейка @1 = «знаю, о чём эта часть». Микро-ячейка @1 = «знаю конкретный паттерн».

### Роль заполнения

Заполнение ячеек — по роли **EDU.R.007 Конструктор ячеек** (PACK-education). Входы: описание принципа + шаблон MAP.001 + frontier science + каталог методов. Выходы: заполненная ячейка + тестовые вопросы + граф зависимостей.

## 3. Разработка ячеек: пошаговый процесс

### Шаг 1: Исследование принципа

Перед заполнением ячеек — глубоко изучить принцип:

1. Прочитать описание принципа в ZP репо (`~/Github/ZP/principles/ZP.N-*.md`)
2. Найти frontier science через WebSearch (2024-2026)
3. Определить: в каких доменах принцип проявляется? Какие типичные заблуждения?
4. Понять зависимости: какие другие принципы нужны как пререквизиты?

### Шаг 2: Заполнение по шаблону

Шаблон ячейки (8 полей) — из Pack EDU `EDU.MAP.001`:

```yaml
cell:
  principle: <ID>              # ZP.1, FPF.Holon, ...
  principle_name: <название>
  depth: <1-5>                 # Bloom depth
  bloom_level: <Remember|Understand|Apply|Analyze|Create>

  # 1. Can-do (3-5 утверждений)
  can_do:
    - "<НАБЛЮДАЕМОЕ поведение>"  # Не «знает», а «может сделать»

  # 2. Методы
  methods:
    primary: [EDU.M.xxx]       # Из Pack EDU
    meta: [EDU.M.006, EDU.M.007]  # Retrieval + Spaced (ВСЕГДА)
    notes: "<специфика>"

  # 3. Пререквизиты
  prerequisites:
    same_principle: <P@(D-1)>  # Предыдущая глубина
    cross_principle: [<список>] # Другие принципы
    cognitive: "<когнитивный уровень>"

  # 4. Оценка
  assessment:
    type: <observation|artifact|test|portfolio|peer_teaching>
    criteria: "<что наблюдаем>"
    transfer_test: "<задание в НОВОМ домене>"
    retrieval_test: "<вопрос через 1+ неделю>"

  # 5. Типичные ошибки (2-4 штуки)
  common_errors:
    - "<ошибка>: <почему возникает>"

  # 6. Домены (≥3, отличаются от предыдущей глубины)
  domains: [<домен1>, <домен2>, <домен3>]

  # 7. Формы подачи (по когнитивному уровню)
  forms:
    preoperational: "<для 3-6>"
    concrete_operational: "<для 7-10>"
    formal_operational: "<для 11-17>"
    postformal: "<для 17+>"

  # 8. Bridge (связь с предыдущей глубиной)
  bridge_template: "Помнишь, как мы <D-1>? Теперь <D>."
```

### Шаг 3: Правила качества

**Can-do утверждения:**
- Начинаются с «Могу...» (наблюдаемое действие)
- НЕ «знает», «понимает», «осознаёт» — это ненаблюдаемо
- Каждое can-do проверяемо за <5 мин (в тесте или наблюдении)
- 3-5 штук на ячейку. Слишком мало = нечем оценивать. Слишком много = переусложнение

**Методы:**
- ВСЕГДА включать EDU.M.006 (Retrieval) и EDU.M.007 (Spaced) — мета-принципы Level 1
- Primary — выбирать по таблице из FORM.006 (глубина → методы)
- Notes — объяснить, ПОЧЕМУ этот метод для ЭТОГО принципа на ЭТОЙ глубине

**Пререквизиты:**
- `same_principle`: всегда P@(D-1), кроме D=1 (null)
- `cross_principle`: только если ДЕЙСТВИТЕЛЬНО блокирует. Не добавлять ради полноты
- `cognitive`: связан с формами подачи. Если cognitive=concrete_operational, то preoperational формы = "Не рекомендуется"

**Assessment:**
- `transfer_test` ОБЯЗАТЕЛЕН — без него неизвестно, произошёл ли перенос (EDU.D.010)
- `retrieval_test` ОБЯЗАТЕЛЕН — без него неизвестно, прочно ли усвоение
- Тип assessment растёт с глубиной: observation → artifact → portfolio → peer_teaching

**Домены:**
- Минимум 3 домена для ротации
- ОТЛИЧАЮТСЯ от предыдущей глубины (для переноса)
- На глубинах 4-5: один домен = собственный профессиональный домен ученика

**Формы:**
- Для каждого когнитивного уровня — конкретная активность с длительностью
- Если глубина слишком высока для уровня — «Не применимо» или «Не рекомендуется»
- Не все глубины доступны на всех когнитивных уровнях (это нормально)

### Шаг 4: Валидация ячейки

#### A. Формальный чеклист (все поля на месте)

- [ ] Все 8 полей заполнены
- [ ] Can-do начинаются с «Могу» и наблюдаемы
- [ ] Methods включают M.006 + M.007
- [ ] Prerequisites непротиворечивы (не требует более высокой глубины, чем текущая)
- [ ] Assessment содержит transfer_test и retrieval_test
- [ ] Домены ≥3 и отличаются от D-1
- [ ] Forms специфичны (с длительностью), не общие
- [ ] Bridge_template связывает с D-1 (кроме D=1)
- [ ] ZP-специфика отражена (не generic)

#### B. Содержательная рубрика CQR (качество заполнения)

> Source-of-truth: EDU.MAP.001 § Cell Quality Rubric

6 измерений, каждое 0/1/2. **Gate: ≥ 9/12, ни одного нуля.**

| # | Измерение | 0 (fail) | 1 (weak) | 2 (strong) |
|---|-----------|----------|----------|------------|
| 1 | Специфичность can-do | Ненаблюдаемое | Наблюдаемое, но широкое | Наблюдаемое + тестируемое за <5 мин |
| 2 | Обоснование метода | Метод без notes | Notes generic | Notes объясняют ПОЧЕМУ для ЭТОГО принципа на ЭТОЙ глубине |
| 3 | Валидность переноса | Нет transfer_test | Transfer в смежном домене | Transfer в принципиально новом домене |
| 4 | Покрытие ошибок | Нет ошибок | Generic ошибки | Когнитивные паттерны + ПОЧЕМУ возникают |
| 5 | Конкретность форм | «Обсуждение» | Активность + длительность | Активность + длительность + материалы |
| 6 | Когерентность bridge | Нет (кроме D=1) | Generic связка | Навык D-1 → действие D |

Записать итоговый балл в конце файла ячейки: `CQR: X/12`

## 4. Разработка тестовых вопросов

Для каждого принципа — банк вопросов для адаптивного теста (EDU.WP.005):

```yaml
question:
  principle: ZP.1
  depth: 3          # Какую глубину проверяет
  type: apply       # Соответствует Bloom
  can_do_ref: "ZP.1@3 can_do[1]"  # Какой can-do проверяет

  prompt: "<текст вопроса>"
  correct_indicators: ["<что ищем в ответе>"]
  scoring: "<binary|rubric>"

  # Варианты (для closed-ended)
  options: [A, B, C, D]
  correct: B

  # Или open-ended
  rubric: "<критерии оценки открытого ответа>"
```

Правила:
- Минимум 3 вопроса на принцип × глубину (для адаптивного теста)
- Каждый вопрос привязан к can-do из ячейки
- Глубины 1-2: закрытые вопросы (выбор). Глубины 3-5: открытые (анализ, создание)

## 5. Разработка программ

Программа = выборка ячеек + форма + расписание:

```yaml
program:
  name: "<название>"
  audience: "<кто>"
  strategy: breadth-first | bottleneck-first
  duration: "<длительность>"

  cells:
    - { principle: ZP.1, depths: [1, 2] }
    - { principle: ZP.2, depths: [1, 2] }
    # ...

  form: concrete_operational  # Какую форму брать из ячеек
  schedule: "<частота и длительность сессий>"
```
